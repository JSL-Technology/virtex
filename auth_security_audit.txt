================================================================================
1. RESUMEN EJECUTIVO
================================================================================
El repositorio muestra una intención de diseño de seguridad avanzada: cookies httpOnly para tokens, rotación de refresh tokens, revocación por tokenVersion, Argon2, MFA/TOTP, passkeys/WebAuthn, CSRF double-submit, validación global de DTOs, guards globales y auditoría de eventos. Sin embargo, NO está listo para producción empresarial al nivel de Auth0/GitHub/Stripe porque existen debilidades críticas y altas en autorización de roles, CSRF incompleto, refresh token sin validación de jti/userId/tokenVersion/expiración de DB, endpoints sensibles sin validación DTO fuerte y exposición innecesaria de access tokens al cliente. La arquitectura tiene buenas bases, pero mezcla decisiones sólidas con bypasses estructurales: el controlador de roles permite CRUD completo sólo con autenticación, varios endpoints mutantes no usan CsrfGuard, el frontend conserva tokens de reseteo/invitación en query params y hay inconsistencias de rutas de usuarios. La remediación exige cambios arquitectónicos, no sólo parches cosméticos: política RBAC obligatoria para roles, refresh tokens opacos o hash server-side, CSRF global para métodos mutantes, validación estricta de todos los cuerpos de auth y saneamiento del modelo de sesión.

Veredicto directo: NO APTO para producción empresarial hasta cerrar los hallazgos Críticos y Altos descritos en este informe.

H1. CRÍTICO — RolesController permite administración de roles sin permisos específicos ni CSRF
--------------------------------------------------------------------------------
Evidencia:
Archivo: apps/backend/api/src/app/roles/roles.controller.ts:10-43
Código actual:
  @Controller('roles')
  @UseGuards(JwtAuthGuard)
  export class RolesController {
    @Get('available-permissions') getAvailablePermissions() { return ALL_PERMISSIONS; }
    @Post() create(...) { return this.rolesService.create(createRoleDto, user.organizationId); }
    @Post('clone/:id') clone(...) { return this.rolesService.cloneRole(id, user.organizationId); }
    @Get() findAll(...) { return this.rolesService.findAllByOrg(user.organizationId); }
    @Patch(':id') update(...) { return this.rolesService.update(id, updateRoleDto, user.organizationId); }
    @Delete(':id') remove(...) { return this.rolesService.remove(id, user.organizationId); }

Explicación:
Cualquier usuario autenticado puede listar permisos disponibles, listar roles, crear roles con permisos arbitrarios, clonar roles, modificar roles y eliminar roles si conoce/obtiene IDs dentro de su organización. En un sistema empresarial, la administración de roles es una operación privilegiada y debe requerir permisos granulares y normalmente step-up/MFA. Además los métodos mutantes usan cookies de sesión y no aplican CsrfGuard.

Corrección propuesta:

  @UseGuards(JwtAuthGuard, PermissionsGuard)
  export class RolesController {

    @HasPermission(PERMISSIONS.ROLES_CREATE)
    create(@Body() dto: CreateRoleDto, @CurrentUser() user: AuthenticatedUser) {
      return this.rolesService.create(dto, user.organizationId);
    }

    @Patch(':id')
    @UseGuards(CsrfGuard, TwoFactorVerifiedGuard)
    @HasPermission(PERMISSIONS.ROLES_EDIT)
    update(@Param('id', ParseUUIDPipe) id: string, @Body() dto: UpdateRoleDto, @CurrentUser() user: AuthenticatedUser) {
      return this.rolesService.update(id, dto, user.organizationId);
    }

    @Delete(':id')
    @UseGuards(CsrfGuard, TwoFactorVerifiedGuard)
    @HasPermission(PERMISSIONS.ROLES_DELETE)
    remove(@Param('id', ParseUUIDPipe) id: string, @CurrentUser() user: AuthenticatedUser) {
      return this.rolesService.remove(id, user.organizationId);
    }
  }
Adicional: validar que permisos asignados no excedan los permisos del actor salvo super-admin explícito.

Referencia:

H2. ALTA — Refresh token no valida completamente vínculo DB-token-usuario-versión-expiración
--------------------------------------------------------------------------------
Evidencia:
Archivo: apps/backend/api/src/app/auth/services/session.service.ts:94-205
Código actual:
  const payload = this.jwtService.verify<JwtPayload & { jti?: string }>(token, { secret: this.configService.get<string>('JWT_REFRESH_SECRET') });
  const user = await this.usersService.findUserByIdForAuth(payload.id);
  if (payload.jti) {
    const refreshTokenEntity = await this.refreshTokenRepository.findOne({
      where: { id: payload.jti },
      select: ['id', 'isRevoked', 'revokedAt', 'replacedByToken', 'userAgent', 'ipAddress', 'userId']
    });
    ...
    refreshTokenEntity.isRevoked = true;
    refreshTokenEntity.revokedAt = new Date();
    await this.refreshTokenRepository.save(refreshTokenEntity);
  }
  const authResponse = await this.tokenService.generateAuthResponse(user, {}, ipAddress, sanitizedUserAgent);

Problemas concretos:
- No exige jti: si un refresh JWT firmado no trae jti, el bloque se omite y se emite una sesión nueva.
- No verifica refreshTokenEntity.userId === payload.id.
- No selecciona ni valida refreshTokenEntity.expiresAt; depende sólo de exp JWT, no del registro server-side.
- No compara payload.tokenVersion contra user.security.tokenVersion, a diferencia de access tokens.
- No almacena hash del refresh token; si JWT_REFRESH_SECRET se compromete, los registros jti activos pueden convertirse en refresh tokens válidos.

Corrección propuesta:
  const payload = this.jwtService.verify<JwtPayload & { jti: string }>(token, {
    secret: this.configService.getOrThrow<string>('JWT_REFRESH_SECRET'),
    algorithms: ['HS256'],
  });
  if (!payload.jti) throw new UnauthorizedException(AuthError.REFRESH_TOKEN_INVALID);

  const user = await this.usersService.findUserByIdForAuth(payload.id);
  if (!user || user.status !== UserStatus.ACTIVE) throw new UnauthorizedException(AuthError.USER_INACTIVE);
  if ((user.security?.tokenVersion ?? 0) !== payload.tokenVersion) throw new UnauthorizedException(AuthError.SESSION_EXPIRED);

  const tokenHash = sha256(token);
  const rt = await this.refreshTokenRepository.findOne({ where: { id: payload.jti, userId: payload.id } });
  if (!rt || rt.tokenHash !== tokenHash || rt.expiresAt <= new Date() || rt.isRevoked) {
    await this.revokeTokenFamily(user.id);
    throw new UnauthorizedException(AuthError.REFRESH_TOKEN_REVOKED);
  }
  // Rotate atomically in transaction with row lock.

Referencia:
OWASP JWT Cheat Sheet; OAuth 2.0 Security Best Current Practice (refresh token rotation/reuse detection); RFC 7519; CWE-287 Improper Authentication.

Explicación:

- En endpoints WebAuthn registrados, usar @UseGuards(JwtAuthGuard, CsrfGuard).
- En checkout, usar @UseGuards(JwtAuthGuard, CsrfGuard) y DTO validado.
- En send-public-verification/verify-public-code, añadir ThrottlerGuard, GoogleRecaptchaGuard y límites por target/IP.

Referencia:

6. HOJA DE RUTA DE REMEDIACIÓN PRIORIZADA
================================================================================

FASE 1 — Crítico (antes de cualquier deploy a producción)
1. Bloquear administración de roles con PermissionsGuard, permisos granulares, CsrfGuard y step-up MFA. Esfuerzo: Medio.
2. Reescribir validación de refresh token: exigir jti, comparar userId, tokenVersion, expiresAt DB, hash de token y rotación atómica transaccional. Esfuerzo: Alto.
3. Aplicar CSRF global para métodos mutantes autenticados y revisar exenciones explícitas. Esfuerzo: Medio.
4. Eliminar accessToken de respuestas browser; dejar sólo cookies httpOnly + user DTO. Esfuerzo: Bajo/Medio.
5. Añadir throttling/recaptcha/attempt counters para send-public-verification y verify-public-code. Esfuerzo: Medio.

FASE 2 — Importante (próximo sprint)
1. Migrar reset/invitación para no depender de query params persistentes; limpiar URL inmediatamente o canjear por cookie temporal httpOnly. Esfuerzo: Medio.
2. Crear DTOs con class-validator para todos los cuerpos de auth/MFA/WebAuthn/checkout. Esfuerzo: Medio.
3. Separar permisos administrativos de usuarios: users.status.update, users.password.reset, users.force_logout; añadir TwoFactorVerifiedGuard. Esfuerzo: Medio.
4. Hacer Redis obligatorio para throttling/cache de auth en producción. Esfuerzo: Bajo.
5. Incrementar tokenVersion o versionar permisos al modificar roles. Esfuerzo: Medio/Alto.
6. Minimizar IP/userAgent/email en logs y auditoría. Esfuerzo: Medio.

FASE 3 — Calidad y escalabilidad (deuda técnica planificada)
1. Eliminar métodos de usuarios del AuthService frontend o moverlos a UsersService. Esfuerzo: Bajo.
2. Añadir permissionsGuard a todas las rutas frontend sensibles de settings. Esfuerzo: Bajo.
3. Revisar almacenamiento de IP/geolocalización: cifrado, truncado, TTL y política de retención. Esfuerzo: Medio.
4. Estandarizar errores de invitación/reset y mensajes i18n sin enumeración. Esfuerzo: Bajo.
5. Añadir pruebas e2e de abuso: rol sin permiso no puede crear roles, CSRF faltante falla, refresh reuse invalida familia, tokenVersion bloquea refresh. Esfuerzo: Alto.
6. Revisar WebAuthnService específico para challenge replay, origin/RP ID, user verification y expiración de challenge. Esfuerzo: Medio.

7. NOTAS DE ALCANCE Y OBJETIVIDAD
================================================================================
- No se auditaron módulos de facturación, inventario, reportes ni UI general salvo cuando impactan directamente auth (por ejemplo create-checkout-session dentro de AuthController por usar sesión/cookies).
- Cuando una sección está bien implementada se reconoció explícitamente; por ejemplo cookies httpOnly, Argon2, tokenVersion, validación global, passkeys y refresh rotation.
- Las severidades no se inflaron: los hallazgos de query params y logging son altos/bajos según impacto y precondiciones; el CRUD de roles es crítico por escalación directa de privilegios.
- Esta auditoría se basa exclusivamente en evidencia visible del repositorio con rutas y líneas citadas.
