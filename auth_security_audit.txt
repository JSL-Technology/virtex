Auditor: Ingeniero senior de seguridad / arquitecto fullstack
Alcance: exclusivamente frontend y backend relacionado con autenticación, autorización, usuarios, roles, tokens y sesiones.

================================================================================
1. RESUMEN EJECUTIVO
================================================================================

El módulo NO está listo para producción a escala empresarial bajo estándares comparables a Auth0, GitHub, Stripe o Google. Hay una base técnica razonable en varios puntos críticos: cookies httpOnly para tokens, expiraciones cortas, argon2, invalidación por tokenVersion, rate limiting global, Helmet, caché de usuario con fallback y rotación de refresh tokens con detección de reutilización. Sin embargo, existen fallas de autorización y sesión que impiden una aprobación empresarial: endpoints de roles carecen de RBAC/CSRF, múltiples mutaciones de usuarios carecen de CSRF/step-up, hay un IDOR en lectura de usuarios por ID, logout actual no revoca necesariamente la sesión porque el sessionId no se propaga desde TokenService, y la contraseña del honeypot produce cookies con tokens falsos. Además, varias decisiones reducen la robustez real frente a XSS, CSRF avanzado, replay y operación multi-instancia: secretos fallback en configuración, CSRF double-submit sin firma, refresh por GET, IP/UA en claro y validaciones de roles/permiso inconsistentes. La conclusión objetiva es: la arquitectura tiene componentes correctos, pero requiere remediación de autorización, CSRF, revocación de sesiones y endurecimiento de secretos antes de cualquier despliegue productivo serio.

F1. Tokens de sesión en cookies httpOnly y refresh cookie con path limitado.
Archivo/líneas: apps/backend/api/src/app/auth/services/cookie.service.ts:24-45
Código actual:

    httpOnly: true,
    secure: isProduction,

    path: '/',
  };
  res.cookie('access_token', accessToken, { ...cookieOptions, maxAge: AuthConfig.COOKIE_ACCESS_MAX_AGE });

H1 — CRÍTICA — Roles sin autorización granular ni CSRF
Evidencia:
Archivo: apps/backend/api/src/app/roles/roles.controller.ts:10-42
Código actual:
  @Controller('roles')
  @UseGuards(JwtAuthGuard)
  export class RolesController {
    @Post()
    create(@Body() createRoleDto: CreateRoleDto, @CurrentUser() user: jwtPayloadInterface.JwtPayload) {
      return this.rolesService.create(createRoleDto, user.organizationId);
    }
    @Patch(':id')
    update(...){ return this.rolesService.update(id, updateRoleDto, user.organizationId); }
    @Delete(':id')
    remove(...){ return this.rolesService.remove(id, user.organizationId); }
  }
Explicación:
Sólo exige autenticación. Cualquier usuario autenticado puede crear roles con permisos arbitrarios, clonar roles, listar permisos disponibles, editar roles y eliminarlos. Como se usan cookies httpOnly, POST/PATCH/DELETE también deben estar protegidos contra CSRF. Esto es una escalación de privilegios directa.
Corrección propuesta:
  @Controller('roles')
  @UseGuards(JwtAuthGuard, PermissionsGuard)
  export class RolesController {
    @Get('available-permissions')
    @HasPermission(PERMISSIONS.ROLES_VIEW)
    getAvailablePermissions() { return ALL_PERMISSIONS; }

    @Post()
    @UseGuards(CsrfGuard, TwoFactorVerifiedGuard)
    @HasPermission(PERMISSIONS.ROLES_CREATE)
    create(@Body() dto: CreateRoleDto, @CurrentUser() user: AuthenticatedUser) {
      return this.rolesService.create(dto, user.organizationId);
    }

    @Patch(':id')
    @UseGuards(CsrfGuard, TwoFactorVerifiedGuard)
    @HasPermission(PERMISSIONS.ROLES_EDIT)
    update(@Param('id', ParseUUIDPipe) id: string, @Body() dto: UpdateRoleDto, @CurrentUser() user: AuthenticatedUser) {
      return this.rolesService.update(id, dto, user.organizationId, user);
    }

    @Delete(':id')
    @UseGuards(CsrfGuard, TwoFactorVerifiedGuard)
    @HasPermission(PERMISSIONS.ROLES_DELETE)
    remove(@Param('id', ParseUUIDPipe) id: string, @CurrentUser() user: AuthenticatedUser) {
      return this.rolesService.remove(id, user.organizationId);
    }
  }
Referencia: OWASP API1:2023 Broken Object Level Authorization; OWASP API5:2023 Broken Function Level Authorization; OWASP CSRF Prevention Cheat Sheet; CWE-862 Missing Authorization; CWE-352 CSRF.
