1. RESUMEN EJECUTIVO
════════════════════════════════════════════════════════════════════════════════

El módulo de autenticación de Virteex ERP presenta un nivel de madurez notable
para un producto en desarrollo activo. La base arquitectónica está bien elegida:
Argon2id para contraseñas, JWT en cookies httpOnly con path restringido, CSRF
por doble-submit cookie, rotación de refresh tokens con detección de reutilización
(nuclear family invalidation), rate limiting, reCAPTCHA v3, 2FA con TOTP + SMS
+ códigos de respaldo argon2-hasheados, y un sistema de auditoría de eventos.
Estas elecciones reflejan un conocimiento real de las mejores prácticas de la
industria.

Sin embargo, se identificaron tres hallazgos de severidad Alta que afectan
directamente la funcionalidad de flujos críticos:

  1. Los endpoints `reset-password` y `set-password-from-invitation` tienen
     protección CSRF que bloquea a usuarios no autenticados o nuevos, lo que
     rompe ambos flujos por completo en el caso común.

  2. El endpoint `POST /auth/reset-password` serializa el objeto `user` crudo
     (con la relación `security` cargada), exponiendo el hash de contraseña,
     el secreto 2FA cifrado y los hashes de códigos de respaldo del propio
     usuario en la respuesta HTTP.

  3. El login via WebAuthn (passkeys) omite la verificación de 2FA para usuarios
     que tienen 2FA activado, creando un bypass del segundo factor.

El módulo NO está listo para producción a escala empresarial sin resolver los
tres hallazgos de severidad Alta y los cuatro de Media. El estado actual es el
de un sistema muy bien intencionado con defectos de implementación puntuales
que se corrigen fácilmente.

────────────────────────────────────────────────────────────────────────────────
HALLAZGO H-01 | ALTA | Fuga de datos sensibles en resetPassword
────────────────────────────────────────────────────────────────────────────────

Archivo: apps/backend/api/src/app/auth/auth.controller.ts:304-308
Archivo: apps/backend/api/src/app/auth/services/password-recovery.service.ts:91

CÓDIGO ACTUAL (auth.controller.ts:304-308):
    async resetPassword(@Body() resetPasswordDto: ResetPasswordDto) {
        const user = await this.passwordRecoveryService.resetPassword(resetPasswordDto);
        const { passwordHash, ...userResult } = user;  // ← INEFECTIVA
        return userResult;  // ← Retorna user.security completo
    }

CÓDIGO ACTUAL (password-recovery.service.ts:60-92):
    const user = await this.userRepository.findOne({
        where: { security: { passwordResetToken: hashedToken } },
        relations: ['security'],  // ← security cargado
    });
    // ...
    const updatedUser = await this.userRepository.save(user);
    return updatedUser;  // ← Incluye updatedUser.security

PROBLEMA:
  La entidad User NO tiene un campo `passwordHash` en su nivel raíz — ese campo
  está en la relación `user.security`. Por tanto, `const { passwordHash, ...userResult } = user`
  desestructura un campo que no existe (undefined), y `userResult` es prácticamente
  idéntico a `user`. La relación `security` cargada en memoria sigue presente
  en `userResult` y se serializa completa en la respuesta HTTP, exponiendo:
    - `security.passwordHash` — hash Argon2 de la nueva contraseña
    - `security.twoFactorSecret` — secreto TOTP cifrado (AES-256-GCM)
    - `security.backupCodes` — array JSON de hashes Argon2 de códigos de respaldo
    - `security.tokenVersion`, `security.failedLoginAttempts`, `security.lockoutUntil`

  Nota de alcance: el atacante necesita un token de reset válido (32 bytes
  aleatorios), por lo que sólo puede ver sus propios datos. Aún así, exponer
  el secreto 2FA cifrado y los hashes de backup codes es innecesario y reduce
  la seguridad en profundidad.

CORRECCIÓN PROPUESTA (auth.controller.ts:304-308):
    async resetPassword(@Body() resetPasswordDto: ResetPasswordDto) {
        const user = await this.passwordRecoveryService.resetPassword(resetPasswordDto);
        return plainToInstance(UserResponseDto, user, { excludeExtraneousValues: true });
    }

REFERENCIA: OWASP API Security Top 10 2023 — API3:2023 Broken Object Property
Level Authorization. CWE-200: Exposure of Sensitive Information to an Unauthorized
Actor. Stripe y Auth0 devuelven únicamente el objeto User sanitizado tras resetear
contraseña, nunca la entidad de seguridad interna.

  El XSRF-TOKEN cookie se genera únicamente como efecto secundario de:
    - Login exitoso (setAuthCookies)
    - Login con 2FA pendiente (setCsrfCookie explícito)
    - Registro exitoso (setAuthCookies)
    - Token refresh (setAuthCookies)

  Ninguna de estas acciones ocurre en el flujo de reset de contraseña:
    1. Usuario olvida contraseña → POST /forgot-password (sin CSRF guard) ✓
    2. Usuario hace click en link del email → GET /auth/reset-password?token=xxx
    3. Usuario envía nueva contraseña → POST /reset-password ← FALLA AQUÍ

  En el paso 3, si el usuario no tiene sesión activa (lo más común cuando se

  XSRF-TOKEN cookie y el guard lanzará ForbiddenException.

  El CSRF guard es innecesario aquí porque el endpoint ya está protegido por
  el token de reset de un solo uso (32 bytes de entropía = 256 bits), que
  actúa como prueba de posesión del email. Agregar CSRF sobre un token de
  un solo uso no añade valor y rompe el flujo legítimo.

CORRECCIÓN PROPUESTA:
  Opción A (recomendada): Eliminar @UseGuards(CsrfGuard) del endpoint.
  El token de reset de 32 bytes es suficiente protección.

    @Public()
    @HttpCode(HttpStatus.OK)
    @UsePipes(new ValidationPipe())

    @Throttle({ default: { limit: 5, ttl: 60000 } })
    async resetPassword(@Body() resetPasswordDto: ResetPasswordDto) { ... }

  Opción B: Agregar un endpoint GET para emitir CSRF token:

    @Public()
    getCsrfToken(@Res({ passthrough: true }) res: Response) {
        this.cookieService.setCsrfCookie(res);
        return { message: 'CSRF token set' };
    }
  Y llamarlo desde el frontend antes del formulario de reset.

REFERENCIA: OWASP CSRF Prevention Cheat Sheet — "Endpoints that are already
  protected by a secret known only to the user (e.g., one-time tokens) do not
  additionally require CSRF protection." GitHub no requiere CSRF en password reset
  ya que el token en el body funciona como synchronizer token.

NOTA-01: El XSRF-TOKEN tiene el mismo maxAge que el access_token (15 min por
  defecto). Dado que el CSRF token no lleva información de autenticación, su
  expiración puede extenderse a 24h sin riesgo de seguridad adicional, lo cual
  reduciría la frecuencia de renovación del token y mejoraría la experiencia
  de usuario en sesiones largas.
  Archivo: cookie.service.ts:58

NOTA-02: El `backup_codes` almacenado como `jsonb` en la misma tabla
  `user_security` implica que cada modificación de backup codes requiere
  una escritura completa del array. Para futuras optimizaciones, considerar
  una tabla `user_backup_codes` con filas individuales y constraint
  UNIQUE(user_security_id, code_hash), que permite operaciones atómicas
  de burn-on-use sin reescribir el array.

NOTA-03: `SET_PASSWORD_FROM_INVITATION` actualmente no limita cuántas veces
  se puede intentar establecer la contraseña (sólo falla si el token expira
  o ya no es PENDING). Aunque el token es de un solo uso implícitamente
  (al usarlo, el estado cambia a ACTIVE), agregar @Throttle explícito
  (3 intentos por token) añade una capa adicional de protección.

NOTA-04: La validación de `organizationId` en el JWT strategy (jwt.strategy.ts:143-164)
  realiza una query adicional a DB para cambios de contexto organizacional.
  Si la arquitectura evoluciona a multi-tenant real, considerar cachear
  la lista de organizaciones del usuario junto con el usuario en Redis.
