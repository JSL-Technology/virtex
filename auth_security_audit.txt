AUDITORÍA DE SEGURIDAD DE AUTENTICACIÓN Y AUTORIZACIÓN
Repositorio: /workspace/virtex
Fecha de auditoría: 2026-06-01
Alcance: exclusivamente autenticación, autorización, usuarios, roles, tokens y seguridad de sesiones en frontend y backend.
Benchmark declarado: prácticas tipo Google/GitHub/Stripe/Auth0, OWASP ASVS, OWASP Top 10, CWE, RFC 6265bis/RFC 7519 y guías de WebAuthn.

================================================================================
1. RESUMEN EJECUTIVO
================================================================================

El repositorio muestra una intención clara de seguridad: cookies httpOnly para tokens, CSRF por double-submit firmado, rotación de refresh tokens, argon2 para contraseñas, invalidación por tokenVersion y controles RBAC/tenant-aware en varias rutas. Sin embargo, NO está listo para producción empresarial porque existen fallos de autorización de alto impacto en el controlador de auth y en operaciones administrativas de roles, inconsistencias frontend/backend que rompen flujos sensibles, y superficies públicas o semi-públicas que deberían estar protegidas explícitamente. El problema más grave es arquitectónico: `AuthController` está marcado completo como público y depende de overrides parciales con `SetMetadata(IS_PUBLIC_KEY, false)`, lo que deja varias rutas sensibles con `@UseGuards(JwtAuthGuard)` pero efectivamente omitidas por el propio guard. También hay debilidades en control de escalación de privilegios, URLs con tokens de recuperación/invitación, validación inconsistente de contraseñas, y eventos/logs con PII.

Veredicto: NO apto para producción a escala empresarial sin remediación previa. La base técnica tiene componentes fuertes, pero los hallazgos críticos/altos impedirían pasar una revisión seria tipo Auth0/GitHub/Stripe.

Evidencia — código actual:

  @UseFilters(TypeOrmExceptionFilter)

Archivo: apps/backend/api/src/app/auth/guards/jwt/jwt.guard.ts:12-22
  canActivate(context: ExecutionContext) {
    const isPublic = this.reflector.getAllAndOverride<boolean>(IS_PUBLIC_KEY, [
      context.getHandler(),
      context.getClass(),
    ]);

    if (isPublic) {
      return true;
    }

    return super.canActivate(context);
  }

Archivo: apps/backend/api/src/app/auth/auth.controller.ts:352-366,448-460,493-518
  @Post('2fa/generate')
  @UseGuards(JwtAuthGuard, CsrfGuard)
  async generateTwoFactorSecret(@CurrentUser() user: User) { ... }

  @Get('webauthn/register/options')
  @UseGuards(JwtAuthGuard)
  async generateWebAuthnRegistrationOptions(@CurrentUser() user: User) { ... }

  @Get('sessions')
  @UseGuards(JwtAuthGuard)
  async getUserSessions(@CurrentUser() user: User, @Req() req: Request) { ... }

Explicación:
`@Public()` en la clase establece metadata pública para todos los handlers. `JwtAuthGuard` consulta handler y clase; si no existe override handler explícito a false, devuelve `true` antes de ejecutar Passport JWT. En este controlador sólo algunos métodos tienen `@SetMetadata(IS_PUBLIC_KEY, false)` (`logout`, `logout-all`, `status`), pero endpoints sensibles como generación/activación 2FA, WebAuthn registration, sesiones y revocación no tienen override. Esto es Broken Access Control. Aunque algunos métodos puedan fallar por `user` undefined, eso no es un control de seguridad aceptable: una ruta sensible nunca debe depender de crash/undefined para negar acceso.

Corrección recomendada:
Rediseñar para que sólo endpoints públicos sean explícitamente públicos. No marcar el controlador completo como `@Public()`.

Código propuesto:
  @ApiTags('Auth')
  @Controller('auth')
  @UseFilters(TypeOrmExceptionFilter)
  export class AuthController {
    @Public()
    @Post('login')
    async login(...) { ... }

    @Public()
    @Post('register')
    async register(...) { ... }

    @Post('2fa/generate')
    @UseGuards(JwtAuthGuard, CsrfGuard)
    async generateTwoFactorSecret(@CurrentUser() user: User) { ... }

    @Get('webauthn/register/options')
    @UseGuards(JwtAuthGuard)
    async generateWebAuthnRegistrationOptions(@CurrentUser() user: User) { ... }
  }

Adicional: crear test e2e que haga request sin cookie/JWT a `/auth/2fa/generate`, `/auth/webauthn/register/options`, `/auth/sessions` y espere 401/403.

Referencia: OWASP Top 10 A01 Broken Access Control; OWASP ASVS V4 Access Control; CWE-284/CWE-306.

H3 — ALTA — WebAuthn registration carece de CSRF y queda afectado por `@Public()`

Evidencia — código actual:
Archivo: apps/backend/api/src/app/auth/auth.controller.ts:448-460
  @Get('webauthn/register/options')
  @UseGuards(JwtAuthGuard)
  async generateWebAuthnRegistrationOptions(@CurrentUser() user: User) {
    return this.webAuthnService.generateRegistrationOptions(user);
  }

  @Post('webauthn/register/verify')
  @UseGuards(JwtAuthGuard)
  @Throttle({ default: { limit: AuthConfig.THROTTLE_LIMIT, ttl: AuthConfig.THROTTLE_TTL } })

Explicación:
Registrar una passkey es una mutación crítica de MFA/credential binding. Debe requerir sesión autenticada, CSRF en POST, idealmente step-up MFA/reautenticación reciente. Por H1, `JwtAuthGuard` puede no ejecutarse por metadata pública de clase; además `verify` no tiene `CsrfGuard`. Un registro de credential WebAuthn no debe quedar expuesto a CSRF ni a bypass de auth.

Corrección recomendada:
Eliminar `@Public()` de clase, añadir `CsrfGuard` y `TwoFactorVerifiedGuard`/re-auth reciente al verify.

Referencia: W3C WebAuthn Level 2/3 security considerations; OWASP ASVS 2.10 Strong Authenticator Binding; CWE-352.
