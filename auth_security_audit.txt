1. RESUMEN EJECUTIVO
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

El módulo de autenticación presenta un diseño de seguridad sólido y maduro que
supera ampliamente la media de proyectos NestJS + Angular equivalentes. Se
implementan correctamente patrones de industria como argon2id, cookies httpOnly
con prefijos __Host-/__Secure-, rotación de refresh tokens con detección de
reuso, invalidación por token version, circuit breaker en cache Redis, doble
submit CSRF, y WebAuthn/Passkeys. La arquitectura demuestra una conciencia de
seguridad consistente a lo largo del codebase.

Sin embargo, se encontraron dos inconsistencias críticas de diseño donde el
accessToken es devuelto en el cuerpo HTTP de respuesta (violando el propio
principio documentado en el código), además de una vulnerabilidad real de
fuerza bruta en la verificación OTP de teléfono, y un potencial de abuso SMS
en el endpoint send-phone-otp.

Veredicto: El módulo NO está completamente listo para producción a escala
empresarial en su estado actual. Los dos leaks de accessToken y la ausencia de
protección contra brute-force en verifyPhoneOtp deben corregirse antes de
cualquier despliegue. El resto del módulo está a un nivel 7-8/10 respecto a
los estándares de Auth0 o GitHub.

F-01 · ARGON2ID COMO ALGORITMO DE HASHING DE CONTRASEÑAS
  Archivo: apps/backend/api/src/app/auth/services/password.service.ts:9-23
  Código:
    async hash(plain: string): Promise<string> {
        return argon2.hash(plain);
    }
  Por qué es correcto: Argon2id es el ganador de la Password Hashing Competition
  (2015) y la recomendación actual de OWASP y NIST SP 800-63B. Supera a bcrypt
  en resistencia a GPU/ASIC y ataques de canal lateral.
  Referencia: OWASP Password Storage Cheat Sheet, CWE-916.

F-02 · COOKIES HTTPONLY CON PREFIJOS __HOST- Y __SECURE-
  Archivo: apps/backend/api/src/app/auth/services/cookie.service.ts:33-50
  Código:
    res.cookie('__Host-access_token', accessToken, {
      httpOnly: true, secure: true, sameSite: 'lax', path: '/'
    });
    res.cookie('__Secure-refresh_token', refreshToken, {
      httpOnly: true, secure: true, path: '/api/v1/auth/refresh'
    });
  Por qué es correcto: El prefijo __Host- exige Secure + Path=/ + sin Domain,
  el más fuerte de los prefijos de cookie (RFC 6265bis §4.1.3). El refresh token
  está correctamente restringido al path de refresco para minimizar superficie.
  Referencia: RFC 6265bis, OWASP Session Management Cheat Sheet.

F-03 · TOKEN VERSION PARA INVALIDACIÓN GLOBAL DE SESIONES
  Archivo: apps/backend/api/src/app/users/entities/user-security.entity.ts:25-30
           apps/backend/api/src/app/auth/services/session.service.ts:296-309
  Código:
    security.tokenVersion = (security.tokenVersion || 0) + 1;
    await this.userSecurityRepository.save(security);
    await this.userCacheService.clearUserSession(userId);
  Por qué es correcto: Permite invalidar instantáneamente TODOS los JWT en
  circulación sin mantener una blacklist de tokens. Cualquier JWT con tokenVersion
  antiguo es rechazado. Este patrón es equivalente al "token family invalidation"
  de Auth0. Se activa en: cambio de contraseña, logout, reuso de refresh token.
  Referencia: OWASP JWT Cheat Sheet, CWE-613.

F-04 · ROTACIÓN DE REFRESH TOKEN CON DETECCIÓN DE REUSO
  Archivo: apps/backend/api/src/app/auth/services/session.service.ts:124-156
  Código:
    const currentTokenHash = crypto.createHash('sha256').update(token).digest('hex');
    if (!refreshTokenEntity || refreshTokenEntity.isRevoked
        || refreshTokenEntity.tokenHash !== currentTokenHash) {
      // NUCLEAR FAMILY INVALIDATION si fuera del grace period
      await this.revokeTokenFamily(user.id);
    }
  Por qué es correcto: La rotación de refresh token (RTR) previene el robo de
  tokens de larga duración. El hash SHA-256 del token (no el token en sí)
  almacenado en DB previene que un volcado de DB permita usar los tokens. La
  "nuclear family invalidation" sigue exactamente el patrón de Auth0 para
  refresh token rotation (si se detecta reuso fuera del período de gracia, se
  invalidan TODAS las sesiones del usuario).
  Referencia: Auth0 Refresh Token Rotation, RFC 6749 §10.4.

F-05 · DOBLE SUBMIT CSRF CON COOKIE NO-HTTPONLY
  Archivo: apps/backend/api/src/app/auth/guards/csrf.guard.ts:24-40
           apps/backend/api/src/app/auth/services/cookie.service.ts:56-68
  Código (guard):
    const xsrfToken = request.headers['x-xsrf-token'];
    const xsrfCookie = request.cookies['XSRF-TOKEN'];
    if (!xsrfToken || !xsrfCookie || xsrfToken !== xsrfCookie) {
      throw new ForbiddenException('Invalid CSRF Token');
    }
  Por qué es correcto: El patrón Double Submit Cookie con CSPRNG (32 bytes
  crypto.randomBytes) es la implementación de referencia de OWASP. La cookie
  no es HttpOnly (debe ser legible por JS) y se renueva en cada login/refresh.
  El frontend (auth.interceptor.ts:32-35) la lee y la envía automáticamente.
  Referencia: OWASP CSRF Prevention Cheat Sheet, CWE-352.

F-06 · CIRCUIT BREAKER EN VALIDACIÓN JWT (REDIS)
  Archivo: apps/backend/api/src/app/auth/strategies/jwt.strategy/jwt.strategy.ts:48-61
  Código:
    this.circuitBreaker = new CircuitBreaker(
      (key: string) => this.cacheManager.get<CachedUser>(key),
      { timeout: 3000, errorThresholdPercentage: 50, resetTimeout: 30000 }
    );
    this.circuitBreaker.fallback(() => Promise.resolve(null));
  Por qué es correcto: El circuit breaker (usando Opossum) garantiza que una
  caída de Redis no brickee el servicio de autenticación. El fallback a DB
  asegura que los usuarios autenticados sigan operando. Patrón usado por
  Stripe y Netflix en sus sistemas de caché críticos.
  Referencia: Martin Fowler, Circuit Breaker Pattern.

F-07 · DETECCIÓN DE VIAJE IMPOSIBLE
  Archivo: apps/backend/api/src/app/auth/services/security-analysis.service.ts:36-76
  Código:
    const speed = distanceKm / safeTimeDiff;
    if (distanceKm > minDistance && speed > maxSpeed) {
      this.eventEmitter.emit('security.suspicious_travel', { userId, speed, ... });

  Por qué es correcto: Se emite un evento de alerta (no se bloquea el login)

F-08 · TEMPTOKEN EN MEMORIA (SIGNAL DE COMPONENTE) PARA 2FA

H-01 · [CRÍTICO] accessToken expuesto en body HTTP — setPasswordFromInvitation
────────────────────────────────────────────────────────────────────────────

  Archivo: apps/backend/api/src/app/auth/auth.controller.ts, líneas 213-224

  Código actual:
    @Post('set-password-from-invitation')
    async setPasswordFromInvitation(...) {
      const { user, accessToken, refreshToken } =
        await this.authFacade.setPasswordFromInvitation(setPasswordDto);

      this.cookieService.setAuthCookies(res, accessToken, refreshToken);

      return {
        user: plainToInstance(UserResponseDto, user, { excludeExtraneousValues: true }),
        accessToken // <-- PROBLEMA: Token expuesto en body HTTP
      };
    }

  Por qué es un problema:
    Este endpoint devuelve el accessToken en el cuerpo JSON de la respuesta,
    violando explícitamente el principio documentado en línea 161 del mismo
    archivo: "// DO NOT return accessToken in body if using cookies."
    Un atacante que logre un XSS puede leer el body de la respuesta y robar
    el accessToken antes de que sea limpiado de memoria. El token ya está
    disponible vía cookie httpOnly; devolverlo en el body duplica la superficie
    de ataque sin ningún beneficio funcional.
    CWE-200: Exposure of Sensitive Information.

  Corrección propuesta:
    return {
      user: plainToInstance(UserResponseDto, user, { excludeExtraneousValues: true }),
      // accessToken OMITIDO — disponible en cookie __Host-access_token
    } as any;

  Referencia: OWASP Top 10 A02:2021 Cryptographic Failures, CWE-200,
    GitHub Security Lab — "Don't return tokens in HTTP responses when using cookies."

    record.lastAttemptAt = new Date();
