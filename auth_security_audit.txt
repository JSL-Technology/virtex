================================================================================
1. RESUMEN EJECUTIVO
================================================================================

Veredicto: NO está listo para producción a escala empresarial tipo Auth0/GitHub/Stripe sin remediar los hallazgos de Fase 1 y Fase 2.

El repositorio muestra una base de seguridad superior a la media: usa cookies httpOnly para tokens en vez de localStorage, CSRF de doble envío firmado, invalidación por tokenVersion, rotación de refresh tokens, hashing Argon2 para contraseñas y códigos de respaldo, guard global JWT, RBAC explícito y caché/circuit breaker en validación de JWT. Sin embargo, hay deficiencias que no pasarían revisión empresarial: cambio de email de cuenta sin step-up ni verificación previa aun cuando el email es identificador de login; endpoint autenticado que acepta URLs de redirección controladas por el cliente; límites incompletos de longitud de password en reset/invitación; problemas de consistencia de cookies en desarrollo/social auth; fuga de detalles de permisos; y algunas áreas no implementadas o solo stub, como activity log. El diseño está cerca de una arquitectura defendible, pero debe endurecer controles sensibles de cuenta, redirecciones, validación y consistencia operativa antes de producción.

  }
No hay verificación de `user.isEmailVerified` antes de login.
Archivo: `apps/backend/api/src/app/users/dto/update-profile.dto.ts`
Líneas: 25-28
  @IsOptional()
  @IsEmail()
  @MaxLength(254)
  email?: string;

Explicación:
El email es el identificador de autenticación (`findUserForAuth(email)` en líneas 50-60 de `auth.service.ts`). Permitir que una sesión autenticada cambie el email sin reautenticación/step-up y sin confirmar primero el nuevo email crea riesgo de toma o bloqueo de cuenta si una sesión es robada. Además, aunque se pone `isEmailVerified = false`, el login no exige email verificado, por lo que el nuevo email no verificado puede seguir operando como identificador de login. En estándares de Auth0/GitHub/Stripe, cambiar email requiere reautenticación y confirmación del nuevo canal antes de hacerlo primario.

Corrección recomendada:
Rediseñar el flujo. No aplicar `email` directamente en `PATCH /users/profile`. Crear flujo separado:
  1. `POST /users/profile/email-change/request` con `currentPassword` o step-up 2FA.
  2. Generar token de cambio de email con hash en DB y expiración corta.
  3. Enviar confirmación al nuevo email.
  4. `POST /users/profile/email-change/confirm` cambia el email solo tras token válido.
  5. Invalidar sesiones (`tokenVersion++`) tras cambio.

Fragmento corregido propuesto:
  // DTO separado
  export class RequestEmailChangeDto {
    @IsEmail()
    @MaxLength(254)
    newEmail!: string;

    @IsString()
    @IsNotEmpty()
    currentPassword!: string;
  }

  async requestEmailChange(userId: string, dto: RequestEmailChangeDto): Promise<void> {
    const user = await this.usersService.findUserByIdForAuth(userId);
    if (!user?.security?.passwordHash) throw new UnauthorizedException();

    const ok = await this.passwordService.verify(user.security.passwordHash, dto.currentPassword);
    if (!ok) throw new UnauthorizedException('Invalid credentials');

    const existing = await this.userRepository.findOne({ where: { email: dto.newEmail } });
    if (existing) throw new BadRequestException('Email change could not be completed.');

    const raw = randomBytes(32).toString('hex');
    user.security.emailChangeToken = sha256(raw);
    user.security.emailChangeTarget = dto.newEmail;
    user.security.emailChangeExpires = new Date(Date.now() + 15 * 60_000);
    await this.userRepository.save(user);
    await this.mailService.sendEmailChangeConfirmation(dto.newEmail, raw);
  }

Referencia:
OWASP ASVS V2 Authentication y V3 Session Management; OWASP Forgot Password Cheat Sheet (one-time tokens and side-channel confirmation); CWE-620 Unverified Password Change / CWE-287 Improper Authentication.
