╔══════════════════════════════════════════════════════════════════════════╗
║ 1. RESUMEN EJECUTIVO                                                       ║
╚══════════════════════════════════════════════════════════════════════════╝

El módulo de autenticación de Virteex es maduro, está bien estructurado y refleja
varias rondas previas de endurecimiento (se ven marcas H1..H18 de auditorías
anteriores ya corregidas). Las primitivas centrales son sólidas y, en varios
puntos, de nivel empresarial: hashing Argon2id, cookies httpOnly (cero tokens en
localStorage en el frontend), rotación de refresh tokens con detección de reuso e
invalidación de familia, CSRF de doble-envío firmado con HMAC, Helmet con HSTS/CSP,
rate limiting respaldado por Redis, validación de entorno con fail-fast (Joi),
aislamiento multi-tenant en consultas de usuarios, y un guard que evita escalación
de privilegios al CREAR/EDITAR roles.

Sin embargo, NO está listo para un despliegue a escala empresarial tal cual: existe
una vía REAL de escalación de privilegios en la ASIGNACIÓN de roles a usuarios
(`UsersService.updateUser` no valida que el actor pueda otorgar el rol que asigna),
que permite a un usuario con `users:edit` —sin ser administrador— auto-asignarse el
rol ADMINISTRATOR (que posee `*`). A esto se suman hallazgos de severidad media:
vinculación de cuentas OAuth sin verificar el claim `email_verified` del proveedor,
token de invitación almacenado en texto plano (a diferencia del de reseteo, que sí
se hashea), TLS de base de datos con verificación de certificado deshabilitada, una
política ABAC (`IsOrganizationOwnerPolicy`) cuya lógica es incorrecta/inerte, y un
desajuste entre el catálogo de permisos y los permisos sembrados en los roles por
defecto.

VEREDICTO: Apto para producción SÓLO tras corregir la Fase 1 (la escalación de
privilegios es bloqueante). Tras ello, el módulo se ubica claramente por encima de
la media de la industria. Puntuación global: 7.5/10.

║ 3. FORTALEZAS — LO QUE ESTÁ BIEN IMPLEMENTADO                              ║
╚══════════════════════════════════════════════════════════════════════════╝

F-1. Cero tokens en almacenamiento accesible por JS (frontend).
     Archivo: apps/core/client-web/src/app/core/services/auth.ts (todo el servicio)
     Toda llamada usa `withCredentials: true` y el estado se guarda como objeto
     `User` en signals; NUNCA se escriben access/refresh tokens en localStorage.
     Ej. login() líneas 142-172, refreshAccessToken() 121-135.
     Por qué es correcto: elimina el vector XSS→robo de token.
     Ref: OWASP "HTML5 Security / Token Storage Cheat Sheet"; OWASP ASVS 3.5.3.

F-2. Hashing Argon2id + ecualización de tiempos + errores genéricos.
     Archivo: apps/backend/api/src/app/auth/services/password.service.ts:13-23
              apps/backend/api/src/app/auth/auth.service.ts:58-76
     `verifyDummy()` ejecuta Argon2 incluso cuando el usuario no existe y siempre
     se lanza `INVALID_CREDENTIALS` genérico + `simulateDelay()`.
     Por qué: evita enumeración por timing y por mensaje.
     Ref: OWASP Authentication Cheat Sheet; CWE-203/CWE-307; NIST SP 800-63B.

F-3. Rotación de refresh tokens con detección de reuso e invalidación de familia.
     Archivo: apps/backend/api/src/app/auth/services/session.service.ts:106-168
     Si se presenta un jti ya revocado → se incrementa `tokenVersion`, se purga la
     caché y se revocan TODOS los refresh del usuario.
     Por qué: patrón de "refresh token rotation" recomendado.
     Ref: OAuth 2.0 Security BCP (RFC 9700) §4.14.2; Auth0 Refresh Token Rotation.

F-4. Defensa por defecto: JwtAuthGuard y ThrottlerGuard globales.
     Archivo: apps/backend/api/src/app/app.module.ts:232-241
     Todo endpoint requiere JWT salvo que se marque `@Public()` explícitamente
     (guards/jwt/jwt.guard.ts:12-23). Rate limit global 20/60s y 5/60s en auth.
     Ref: principio "secure by default"; OWASP API4:2023 (Rate Limiting).

F-5. CSRF de doble-envío FIRMADO (HMAC) con comparación timing-safe.
     Archivo: apps/backend/api/src/app/auth/services/cookie.service.ts:51-92
              apps/backend/api/src/app/auth/guards/csrf.guard.ts:21-44
     Token = nonce.HMAC(secret,nonce); el guard recomputa el HMAC y usa
     `crypto.timingSafeEqual`. Cookies httpOnly + Secure(prod) + SameSite=Lax.
     Ref: OWASP CSRF Cheat Sheet (Signed Double-Submit Cookie).

F-6. Hardening de cabeceras y CORS con allowlist.
     Archivo: apps/backend/api/src/main.ts:38-77
     Helmet con HSTS (1 año, includeSubDomains, preload) en prod, CSP sin
     unsafe-inline en prod, frameAncestors 'none', objectSrc 'none'; CORS por
     lista configurable (no comodín) con credentials.
     Ref: OWASP Secure Headers Project; CWE-1021 (clickjacking).

F-7. Validación de entorno fail-fast para secretos.
     Archivo: apps/backend/api/src/app/app.module.ts:85-104 (Joi),
              apps/backend/api/src/app/auth/auth.config.ts:6-17 (requireSecret)
     JWT_SECRET / JWT_REFRESH_SECRET / CSRF_SECRET / ENCRYPTION_SECRET exigen
     min 32 chars; valores tipo "change_me/default/secret" abortan el arranque.
     Ref: CWE-798 (credenciales embebidas); 12-Factor App.

F-8. Aislamiento multi-tenant en consultas de usuario (anti-IDOR).
     Archivo: apps/backend/api/src/app/users/users.service.ts:218-227 (findOneByOrg),
              users.controller.ts:158-165, 216-226; ParseUUIDPipe en todos los :id.
     Por qué: previene lectura/edición cruzada entre organizaciones.
     Ref: OWASP API1:2023 (BOLA/IDOR); CWE-639.

F-9. Anti-escalación al crear/editar/clonar roles.
     Archivo: apps/backend/api/src/app/roles/roles.service.ts:31-71
     `assertAssignablePermissions` impide asignar a un rol permisos que el actor no
     posee y prohíbe delegar el comodín `*`. DTO valida cada permiso contra el
     catálogo (`@IsIn(ALL_PERMISSIONS)`, create-role.dto.ts:14-18).
     Ref: CWE-269 (Privilege Management). [Ver H-01: este control NO cubre la
     asignación de roles a usuarios.]

F-10. 2FA robusto.
     Archivo: apps/backend/api/src/app/auth/services/two-factor-auth.service.ts
     - Secreto TOTP cifrado AES-256-GCM en reposo (cryptoUtil.encrypt, líneas 39-43).
     - Backup codes hasheados con Argon2 y "burn-on-use" (159-178).
     - Step-up con contraseña para activar 2FA (98-108).
     Ref: NIST SP 800-63B §5.1.5; OWASP MFA Cheat Sheet.

F-11. WebAuthn correcto.
     Archivo: apps/backend/api/src/app/auth/services/webauthn.service.ts
     userVerification 'required', challenge de un solo uso con TTL (del tras uso,
     líneas 90/157), verificación de counter (clon detection, 151-156).
     Ref: W3C WebAuthn; FIDO2.

F-12. Aislamiento de campos sensibles + DTO de salida con whitelist.
     Archivo: apps/backend/api/src/app/users/entities/user-security.entity.ts
              apps/backend/api/src/app/auth/dto/user-response.dto.ts (@Expose)
     passwordHash, twoFactorSecret, backupCodes, tokens viven en `UserSecurity`
     (entidad separada, no eager) y nunca se exponen; las respuestas usan
     `plainToInstance(..., { excludeExtraneousValues: true })`.
     Ref: OWASP API3:2023 (Excessive Data Exposure); CWE-213.

F-13. Hardening de privacidad en logs: hash de email y enmascarado/cifrado de IP.
     Archivo: token.service.ts:204-235; session.service.ts:210-224
     IP mostrada enmascarada; IP cifrada (AES-GCM, columna select:false) sólo para
     forense; emails hasheados antes de loguear (H9/H14).
     Ref: GDPR minimización; OWASP Logging Cheat Sheet.

[H-01] ALTA — Escalación de privilegios vía asignación de roles a usuarios
────────────────────────────────────────────────────────────────────────────
Capa: Backend
Archivos:
  - apps/backend/api/src/app/users/users.controller.ts:167-182
  - apps/backend/api/src/app/users/users.service.ts:146-181
  - apps/backend/api/src/app/users/dto/update-user.dto.ts:15-18

EVIDENCIA (controller):
  167  @Patch(':id')
  168  @UseGuards(CsrfGuard, TwoFactorVerifiedGuard)
  169  @HasPermission(PERMISSIONS.USERS_EDIT)         // sólo exige 'users:edit'
  ...
  176    const updatedUser = await this.usersService.updateUser(id, updateUserDto, user.organizationId);

EVIDENCIA (service):
  161   const { roleId, ...userData } = updateUserDto;
  165   if (roleId) {
  166     const role = await this.rolesService.findOne(roleId, organizationId);  // sólo valida que el rol exista en el org
  170     user.roles = [role];                                                   // <-- asigna SIN validar al actor
  ...

EVIDENCIA (DTO):
  15  @IsOptional() @IsUUID() roleId?: string;

EXPLICACIÓN:
`assertAssignablePermissions` (roles.service.ts:31-45) protege la CREACIÓN/EDICIÓN
de roles, pero NO la ASIGNACIÓN de un rol existente a un usuario. Todo organización
tiene sembrado el rol ADMINISTRATOR con permisos `['*']` (config/roles.config.ts:4-8).
Un actor con el permiso `users:edit` (pero sin `*`) —un rol "Manager" perfectamente
legítimo que un admin podría crear— puede ejecutar:
   PATCH /users/{suPropioId}  { "roleId": "<id-del-rol-ADMINISTRATOR>" }
y obtener el comodín `*`, comprometiendo toda la organización. Es una escalación
vertical de privilegios real (no requiere ser admin previamente).

CORRECCIÓN (validar que el actor pueda otorgar ese rol):
  // users.service.ts — al inicio de updateUser, recibir el actor
  async updateUser(id, dto, organizationId, actor: AuthenticatedUser) {
    ...
    if (roleId) {
      const role = await this.rolesService.findOne(roleId, organizationId);
      if (!role) throw new NotFoundException(...);
      // 1) El actor sólo puede asignar roles cuyos permisos él ya posee:
      this.rolesService.assertCanAssignRole(actor, role); // reutilizar assertAssignablePermissions(actor, role.permissions)
      // 2) (Defensa en profundidad) impedir delegar '*' salvo super-admin:
      if (role.permissions.includes('*') && !actor.permissions.includes('*'))
        throw new ForbiddenException('No puedes asignar un rol con privilegios totales.');
      user.roles = [role];
      ...
    }
  }
Además, exponer `assertAssignablePermissions` como método público de RolesService y
pasar `@CurrentUser()` desde el controller (hoy `updateUser` no recibe el actor).

REFERENCIA: OWASP API3:2023 (Broken Object Property Level Authorization),
CWE-269 (Improper Privilege Management), CWE-266 (Incorrect Privilege Assignment).

[M-02] MEDIA — Vinculación de cuentas OAuth sin verificar email_verified
────────────────────────────────────────────────────────────────────────────
Capa: Backend
Archivo: apps/backend/api/src/app/auth/services/social-auth.service.ts:23-56

EVIDENCIA:
  25   const user = await this.usersService.findUserForAuth(socialUser.email);
  27   if (user) {
  28     if (user.authProvider !== socialUser.provider || user.authProviderId !== socialUser.providerId) {
  29       await this.usersService.update(user.id, {
  30         authProvider: socialUser.provider,
  31         authProviderId: socialUser.providerId,   // <-- se (re)vincula automáticamente por coincidencia de email
  ...

EXPLICACIÓN:
Cuando llega un login social cuyo email coincide con una cuenta existente, se vincula
y se SOBRESCRIBE `authProvider`/`authProviderId` con lo que entregue el IdP, sin
comprobar el claim `email_verified` del proveedor. Con Google/Microsoft sobre dominios
gestionados suele venir verificado, pero con proveedores OIDC/Okta personalizados o
dominios no verificados, un atacante que controle un IdP capaz de afirmar un email
ajeno podría apropiarse de la cuenta local (pre-account takeover). También permite que
un segundo proveedor "secuestre" el providerId previamente vinculado.

CORRECCIÓN:
  - Verificar el claim de email del IdP antes de vincular:
      if (!socialUser.emailVerified) throw new UnauthorizedException('Email no verificado por el proveedor.');
    (propagar `email_verified` desde cada estrategia: google/microsoft/okta.strategy.ts).
  - No re-vincular automáticamente si ya existe una contraseña local: requerir un paso
    de "vinculación de cuenta" autenticado (login local o verificación por correo).
REFERENCIA: OWASP "Authentication Cheat Sheet" (account linking), CWE-287,
incidente clásico "Pre-account hijacking via OAuth" (Microsoft/Okta research, 2022).

[M-03] MEDIA — Token de invitación almacenado en texto plano
────────────────────────────────────────────────────────────────────────────
Capa: Backend
Archivos:
  - apps/backend/api/src/app/users/users.service.ts:323-335
  - apps/backend/api/src/app/auth/services/password-recovery.service.ts:87-128

EVIDENCIA (generación, en claro):
  323  const invitationToken = crypto.randomBytes(32).toString('hex');
  ...
  334  invitationToken: invitationToken,          // se persiste tal cual

EVIDENCIA (consulta directa por el valor crudo):

  107    where: { invitationToken: token, status: UserStatus.PENDING, invitationTokenExpires: MoreThan(new Date()) },

CONTRASTE — el token de reseteo SÍ se hashea (buena práctica que aquí falta):

REFERENCIA: CWE-522 (Insufficiently Protected Credentials), OWASP "Forgot Password
Cheat Sheet" (los tokens de un solo uso deben guardarse hasheados).

[M-04] MEDIA — TLS de base de datos con verificación de certificado deshabilitada
────────────────────────────────────────────────────────────────────────────

Archivo: apps/backend/api/src/app/app.module.ts:151-153

EXPLICACIÓN:

[M-05] MEDIA — Política ABAC IsOrganizationOwnerPolicy incorrecta/inerte
────────────────────────────────────────────────────────────────────────────
Capa: Backend
Archivos:
  - apps/backend/api/src/app/auth/policies/is-organization-owner.policy.ts:10-30
  - uso: apps/backend/api/src/app/users/users.controller.ts:184-191 (DELETE /users/:id)

EVIDENCIA:
  15   const resourceId = request.params?.id;          // en DELETE /users/:id ESTO ES UN userId
  19   if (user.roles?.some((r) => r.name === 'ADMIN')) return true;   // el rol real es 'ADMINISTRATOR'
  25   if (resourceId && user.organization?.id === resourceId) return true;  // compara userId vs orgId → nunca true
  29   return false;

EXPLICACIÓN:
1) Compara el `:id` de la ruta (un UUID de USUARIO) contra `user.organization.id`
   (UUID de ORGANIZACIÓN): nunca coincide.
2) El nombre de rol comprobado es `'ADMIN'`, pero el rol real es
   `RoleEnum.ADMINISTRATOR` ('ADMINISTRATOR'): nunca coincide.
En consecuencia la política devuelve SIEMPRE false. La única razón por la que DELETE
/users/:id funciona para administradores es que `PermissionsGuard` cortocircuita ANTES
con el comodín `*` (permissions.guard.ts:42-44), de modo que las políticas ABAC ni
siquiera se evalúan para super-admins. Resultado: lógica de seguridad muerta que da
falsa sensación de control de propiedad, y que jamás restringirá realmente a un actor
no-comodín con `users:delete`.

CORRECCIÓN:
  - Corregir el nombre de rol ('ADMINISTRATOR') o, mejor, basarse en permisos.
  - Cargar el recurso real y comparar su `organizationId` con el del actor:
      const target = await this.usersRepo.findOne({ where:{ id: resourceId }, select:['organizationId'] });
      return !!target && target.organizationId === user.organizationId;
  - Reconsiderar que `*` saltee TODAS las policies: para acciones críticas, las
    policies deberían evaluarse incluso para super-admins (o documentarlo explícitamente).
REFERENCIA: CWE-863 (Incorrect Authorization), CWE-561 (Dead Code en ruta de seguridad).

[L-14] BAJA — Rotación de refresh sin bloqueo: falsos positivos de reuso
────────────────────────────────────────────────────────────────────────────
Capa: Backend
Archivo: apps/backend/api/src/app/auth/services/session.service.ts:108-168

EXPLICACIÓN:
Entre la lectura `findOne({where:{id:jti}})` y el `save({isRevoked:true})` no hay
transacción/lock. Dos refresh concurrentes legítimos (común en SPAs con varias
pestañas) pueden ambos ver `isRevoked=false` y, en el segundo, dispararse la
detección de reuso → invalidación de TODA la familia y logout global del usuario
legítimo (denegación de servicio para el propio usuario). El grace period
(`AuthConfig.REFRESH_GRACE_PERIOD`) existe pero no se aplica aquí.

CORRECCIÓN: hacer la rotación atómica:
  - UPDATE ... SET is_revoked=true WHERE id=:jti AND is_revoked=false RETURNING ...;
    si affected=0 → ya fue rotado: aplicar grace period (aceptar si revokedAt < now-grace
    y el sucesor pertenece a la misma familia) antes de declarar reuso.
REFERENCIA: RFC 9700 §4.14.2 (refresh rotation con tolerancia a concurrencia); CWE-362.

================================================================================
1. RESUMEN EJECUTIVO
================================================================================

Veredicto: el módulo NO está listo para producción a escala empresarial con el estándar de Auth0, GitHub o Stripe.

El backend contiene varias decisiones sólidas: cookies httpOnly para tokens, CSRF firmado de doble envío, JWT de acceso RS256 con kid, refresh tokens hasheados y rotación transaccional, validación global estricta y redacción de secretos en logs. Sin embargo, existen defectos funcionales y de seguridad que impedirían aprobarlo en un comité de seguridad empresarial: el endpoint /auth/status verifica access tokens con JWT_SECRET aunque los access tokens se emiten en RS256; el logout frontend usa sendBeacon sin header CSRF y puede no revocar la sesión en servidor; la edición administrativa de usuarios permite cambiar emails sin flujo de verificación; el frontend no envía currentPassword al habilitar 2FA aunque el backend lo exige; y el DTO OpenAPI aún documenta tokens en cuerpo pese a que el contrato real dice cookie-only.

La arquitectura muestra intención de defensa en profundidad, pero todavía hay inconsistencias entre frontend y backend y problemas de ciclo de vida de sesión que son críticos en autenticación empresarial. Las áreas de mayor riesgo son consistencia de sesión, cambios de identidad/email, step-up MFA y contratos públicos de API.

3. FORTALEZAS — LO QUE ESTÁ BIEN IMPLEMENTADO
================================================================================

F1 — Tokens en cookies httpOnly con flags razonables
Archivo: apps/backend/api/src/app/auth/services/cookie.service.ts
Líneas: 21-38
Código actual:
    const baseOptions = {
      httpOnly: true,
      secure: isProduction,
      sameSite: 'lax' as 'strict' | 'lax' | 'none',
      path: '/',
    };

    res.cookie(accessTokenName, accessToken, {
      ...baseOptions,
      maxAge: AuthConfig.COOKIE_ACCESS_MAX_AGE,
    });

Por qué está bien: los tokens no están expuestos a JavaScript, reduciendo exfiltración por XSS. En producción usa prefijos __Host-/__Secure- y secure=true.
Referencia: OWASP Session Management Cheat Sheet; OWASP HTML5 Security Cheat Sheet; RFC 6265bis cookie prefixes.

F2 — CSRF con doble envío firmado y comparación timing-safe
Archivo: apps/backend/api/src/app/auth/guards/csrf.guard.ts
Líneas: 24-66
Código actual:
    const headerValue = request.headers['x-xsrf-token'] as string | undefined;
    const cookieValue = request.cookies['XSRF-TOKEN'] as string | undefined;
    if (!headerValue || !cookieValue || headerValue !== cookieValue) {
      ...
      throw new ForbiddenException('Invalid CSRF Token');
    }
    ...
    const expectedMac = crypto
      .createHmac('sha256', AuthConfig.CSRF_SECRET)
      .update(nonce)
      .digest('base64url');
    ...
    if (expectedBuf.length !== receivedBuf.length || !crypto.timingSafeEqual(expectedBuf, receivedBuf)) {
      ...
      throw new ForbiddenException('Invalid CSRF Token');
    }

Por qué está bien: el patrón firmado evita que una cookie XSRF fijada por subdominio sea aceptada sin conocer el secreto HMAC.
Referencia: OWASP CSRF Prevention Cheat Sheet, Signed Double-Submit Cookie; CWE-352.

Archivo: apps/backend/api/src/app/auth/strategies/jwt.strategy/jwt.strategy.ts
Líneas: 37-67
Código actual:

      ...
      const header = JSON.parse(Buffer.from(parts[0], 'base64url').toString('utf8'));
      const publicKey = keyManagementService.getPublicKey(header.kid);
      ...
    },

Archivo: apps/backend/api/src/app/auth/services/token.service.ts
Líneas: 236-244
Código actual:
    const { kid, privateKey } = this.keyManagementService.getActiveKey();

Explicación:
No es un exploit directo, pero en autenticación la cadena de interceptores y fetch/XSRF debe ser inequívoca. Providers duplicados aumentan riesgo de comportamiento dependiente de orden y dificultan garantías de que todos los requests pasan por authInterceptor.

Corrección propuesta:

      ...CORE_PROVIDERS_SIN_provideHttpClient,
      provideHttpClient(withInterceptors([authInterceptor]), withFetch()),
      provideServiceWorker('ngsw-worker.js', { enabled: !isDevMode(), registrationStrategy: 'registerWhenStable:30000' }),
    ]

Referencia:
