1. RESUMEN EJECUTIVO
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

El módulo de autenticación de Virteex muestra una arquitectura deliberada y
bien pensada: Argon2id para hashing, rotación de refresh tokens con detección
de reuso, versionado de tokens para invalidación de sesiones, CSRF double-submit
cookie, guard JWT global con patrón @Public(), 2FA con TOTP+SMS+Backup Codes, y
WebAuthn. Son decisiones técnicas de nivel empresarial.

Sin embargo, la implementación tiene TRES fallos graves que, en conjunto, hacen
que el módulo NO esté listo para producción a escala empresarial:

(1) El endpoint de logout no ejecuta ninguna lógica de invalidación de sesión
    — solo borra cookies. Un atacante con un refresh token capturado puede
    re-autenticarse indefinidamente después de que el usuario haga logout.

(2) Existe una vulnerabilidad IDOR en GET /users/:id que permite a cualquier
    usuario autenticado consultar el perfil de usuarios de OTRAS organizaciones.

(3) El flujo de reset de contraseña iniciado por un administrador es
    funcionalmente roto e incompatible con el endpoint de validación, lo que
    significa que esa funcionalidad no funciona en absoluto.

Adicionalmente hay un secreto JWT con valor por defecto inseguro en código
fuente para el token temporal de 2FA, que debe considerarse comprometido si
la variable de entorno no está configurada.

Veredicto: NO listo para producción empresarial en su estado actual. Los
problemas críticos son corregibles, pero algunos requieren refactorización de
lógica de servicio, no parches superficiales.

BACKEND
─────────────────────────────────────────────────────────────────────────────
Seguridad:          6/10
  Razón: Argon2id, tokenVersion, detección reuso son excelentes. El logout
  no revoca tokens en DB, IDOR en users/:id, y JWT_2FA secret hardcodeado
  son fallos concretos que bajan la nota significativamente.
  Evidencia: session.service.ts:325-329, users.controller.ts:143-147,
             auth.config.ts:24

Arquitectura:       8/10
  Razón: Excelente separación de responsabilidades (Facade, Services,
  Guards, Strategies, Listeners). El circuit breaker para Redis y el patrón
  de Event Emitter para auditoría son decisiones maduras. Penaliza la
  inconsistencia entre dos flujos de reset de contraseña.
  Evidencia: auth.facade.ts, jwt.strategy.ts (circuit breaker en líneas 45-58)

Validación:         7/10
  Razón: ValidationPipe global con whitelist:true es correcto. DTOs bien
  decorados con class-validator. Sin embargo, ChangePasswordDto carece de
  validadores de complejidad que sí existen en RegisterUserDto y ResetPasswordDto.
  Evidencia: change-password.dto.ts:1-10, register-user.dto.ts:59-63

Manejo de errores:  7/10
  Razón: AuthException custom, mensajes genéricos en forgot-password y login
  (anti-enumeración). Sin embargo, console.error desnudo en users.service.ts:242
  puede exponer datos en ciertos entornos de logging.
  Evidencia: auth.exceptions.ts, password-recovery.service.ts:38, users.service.ts:242

Escalabilidad:      8/10
  Razón: Caching de usuarios en Redis, circuit breaker para fallback a DB,
  throttler con Redis store, token cleanup en batches. Sólido.
  Evidencia: jwt.strategy.ts:45-113, session.service.ts:353-416

Observabilidad:     5/10
  Razón: AuthAuditListener solo hace console.log, no persiste en DB. El
  audit trail para eventos de login (éxito/fallo) no es durable. MfaOrchestratorService
  sí persiste vía AuditTrailService para 2FA, creando inconsistencia.
  Evidencia: auth-audit.listener.ts:1-21, mfa-orchestrator.service.ts:173-195

FRONTEND
─────────────────────────────────────────────────────────────────────────────
Seguridad del cliente:     8/10
  Razón: No hay tokens en localStorage. Se usan httpOnly cookies. El
  interceptor extrae el XSRF-TOKEN de cookie y lo manda en header manualmente.
  El accessToken se elimina de la respuesta antes de exponerlo al componente.
  Evidencia: auth.ts:171-173, auth.interceptor.ts:26-37

Manejo de tokens/sesión:   8/10
  Razón: Refresh automático con cola (AuthQueueService), exponential backoff,
  manejo de 401. La señal reactiva de Angular es correcta. El changePassword
  lleva IS_PUBLIC_API=true inadvertidamente, lo que puede causar logout
  prematuro si el token expira durante la operación.
  Evidencia: auth.interceptor.ts:47-103, auth.ts:521-526

Protección de rutas:       8/10
  Razón: authGuard valida contra backend, no solo señal local. permissionsGuard
  verifica permisos del usuario. Rutas públicas (payment, auth) correctamente
  excluidas del authGuard.
  Evidencia: auth-guard.ts:1-33, app.routes.ts:22-23

Consistencia con backend:  7/10
  Razón: Los guards del frontend son UX, no seguridad. El backend tiene
  JwtAuthGuard global y PermissionsGuard por endpoint. La consistencia es
  razonable, pero el endpoint GET /users/:id no se protege en backend
  contra IDOR, lo que el frontend no puede compensar.
  Evidencia: users.controller.ts:140-147

──────────────────────────────────────────────────────────────────────────────
PUNTUACIÓN GLOBAL: 7/10
  Un 7 refleja que la arquitectura y la mayoría de los controles están bien
  diseñados, pero los fallos en logout, IDOR y el flujo roto de reset de
  contraseña administrativo son lo suficientemente graves para bloquear el
  despliegue en producción.

3. FORTALEZAS — LO QUE ESTÁ BIEN IMPLEMENTADO
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

F1. ARGON2ID CON PARÁMETROS ROBUSTOS
  Archivo: apps/backend/api/src/app/auth/services/password.service.ts:8-23
           apps/backend/api/src/app/auth/auth.config.ts:50-52
  Fragmento:
    async hash(plain: string): Promise<string> {
        return argon2.hash(plain); // Uses ARGON2_MEMORY_COST=65536, TIME_COST=3, PARALLELISM=4
    }
  Por qué es correcto: Argon2id es el ganador del Password Hashing Competition
  (2015) y la recomendación actual de OWASP. Los parámetros (64MB, 3 iter,
  4 threads) superan los mínimos de OWASP.
  Referencia: OWASP Password Storage Cheat Sheet, RFC 9106

F2. VERIFICACIÓN DE CONTRASEÑA DUMMY CONTRA TIMING ATTACKS
  Archivo: apps/backend/api/src/app/auth/auth.service.ts:61-64
  Fragmento:
    } else {
        await this.passwordService.verifyDummy(password);
        isPasswordValid = false;
    }
  Por qué es correcto: Cuando el usuario no existe, se realiza una operación
  de verificación de coste igual al de un usuario real. Esto iguala el tiempo
  de respuesta y previene ataques de temporización que confirmarían la
  existencia de un usuario.
  Referencia: CWE-208 (Observable Timing Discrepancy), OWASP Authentication Cheat Sheet

F3. TOKEN VERSIONING PARA INVALIDACIÓN GLOBAL DE SESIONES
  Archivo: apps/backend/api/src/app/users/entities/user-security.entity.ts:23-30
           apps/backend/api/src/app/auth/strategies/jwt.strategy/jwt.strategy.ts:120-124
  Fragmento:
    @Column({ name: 'token_version', type: 'integer', default: 0,
              comment: 'Incrementado para invalidar todos los JWT emitidos previamente.' })
    tokenVersion: number;
    // En jwt.strategy.ts:
    if (currentTokenVersion !== tokenVersion) {
      throw new UnauthorizedException(AuthError.SESSION_EXPIRED);
    }
  Por qué es correcto: Este patrón permite invalidar TODOS los tokens activos
  de un usuario (cambio de contraseña, bloqueo, cambio de rol) sin necesidad
  de una lista de revocación. Es la misma técnica usada por Google y GitHub.
  Referencia: OWASP JWT Cheat Sheet

F4. ROTACIÓN DE REFRESH TOKENS CON DETECCIÓN DE REUSO Y NUCLEAR INVALIDATION
  Archivo: apps/backend/api/src/app/auth/services/session.service.ts:109-152
  Fragmento:
    if (!refreshTokenEntity || refreshTokenEntity.isRevoked) {
      // Si hay reuso de token revocado → invalidar familia completa
      user.security.tokenVersion = (user.security.tokenVersion || 0) + 1;
      await this.userSecurityRepository.save(user.security);
      await this.refreshTokenRepository.update(
          { userId: user.id, isRevoked: false },
          { isRevoked: true, revokedAt: new Date() }
      );
      throw new UnauthorizedException(AuthError.REFRESH_TOKEN_REVOKED);
    }
  Por qué es correcto: El patrón "Refresh Token Rotation + Family Invalidation"
  es la defensa estándar contra el robo de refresh tokens. RFC 6749 lo menciona
  y Auth0 lo documenta como best practice.
  Referencia: RFC 6749 §10.4, OAuth 2.0 Security Best Current Practice §4.12

F5. CSRF DOUBLE SUBMIT COOKIE CORRECTAMENTE IMPLEMENTADO
  Archivo: apps/backend/api/src/app/auth/guards/csrf.guard.ts:1-44
           apps/backend/api/src/app/auth/services/cookie.service.ts:51-63
  Fragmento (guard):
    const xsrfToken = request.headers['x-xsrf-token'];
    const xsrfCookie = request.cookies['XSRF-TOKEN'];
    if (!xsrfToken || !xsrfCookie || xsrfToken !== xsrfCookie) {
      throw new ForbiddenException('Invalid CSRF Token');
    }
  Por qué es correcto: Genera token con crypto.randomBytes(32), lo almacena
  en cookie no-httpOnly (para que JS lo lea), y verifica que el header enviado
  coincide con la cookie. Correcto y suficiente.
  Referencia: OWASP CSRF Cheat Sheet (Double Submit Cookie Pattern)

F6. CIRCUIT BREAKER PARA REDIS EN JWT STRATEGY
  Archivo: apps/backend/api/src/app/auth/strategies/jwt.strategy/jwt.strategy.ts:44-58
  Fragmento:
    this.circuitBreaker = new CircuitBreaker(
      (key: string) => this.cacheManager.get<CachedUser>(key),
      { timeout: 3000, errorThresholdPercentage: 50, resetTimeout: 30000 }
    );
    this.circuitBreaker.fallback(() => Promise.resolve(null));
  Por qué es correcto: Si Redis falla, el sistema no bloquea las peticiones —
  hace fallback a DB. Esto previene outages de autenticación completos cuando
  el cache no está disponible, cumpliendo con principios de alta disponibilidad.
  Referencia: Principio de Resiliencia / Pattern: Circuit Breaker (Fowler)

F7. AISLAMIENTO MULTI-TENANT EN IMPERSONACIÓN
  Archivo: apps/backend/api/src/app/auth/services/impersonation.service.ts:40-64
  Fragmento:
    if (targetUser.organizationId !== adminUser.organizationId) {
        throw new ForbiddenException('No puedes suplantar usuarios de otra organización.');
    }
    if (targetLevel > adminLevel) {
        throw new ForbiddenException('No tienes jerarquía suficiente...');
    }
  Por qué es correcto: Doble validación (org + jerarquía) para impersonación.
  Previene escalación de privilegios y cross-tenant access.

F8. COOKIES HTTP-ONLY + SECURE + SAMESITE=LAX
  Archivo: apps/backend/api/src/app/auth/services/cookie.service.ts:11-49
  Fragmento:
    const cookieOptions = {
      httpOnly: true,
      secure: isProduction,
      sameSite: 'lax',
      path: '/',
    };
  Por qué es correcto: httpOnly previene acceso JS (XSS). secure activo en
  producción. SameSite=Lax es el balance correcto para OAuth callbacks sin
  sacrificar protección CSRF.
  Referencia: RFC 6265bis, OWASP Session Management Cheat Sheet

F9. BACKUP CODES DE 2FA HASHEADOS CON ARGON2
  Archivo: apps/backend/api/src/app/auth/services/two-factor-auth.service.ts:166-178
  Fragmento:
    const code = crypto.randomBytes(4).toString('hex').toUpperCase();
    const formattedCode = `${code.slice(0, 4)}-${code.slice(4)}`;
    codes.push(formattedCode);
    hashedCodes.push(await argon2.hash(formattedCode));
  Por qué es correcto: Los backup codes son equivalentes a contraseñas —
  hashearlos con Argon2 es la práctica correcta. Uso único implementado
  correctamente (burn on use).
  Referencia: OWASP MFA Cheat Sheet

F10. SECRETO TOTP CIFRADO CON AES-256-GCM
  Archivo: apps/backend/api/src/app/auth/services/two-factor-auth.service.ts:37-42
  Fragmento:
    const encryptedSecret = this.cryptoUtil.encrypt(secret);
    security.twoFactorSecret = encryptedSecret;
  Por qué es correcto: El secreto TOTP es tan sensible como una contraseña.
  Cifrarlo en reposo con AES-256-GCM (autenticado) limita el impacto de una
  brecha de base de datos.
  Referencia: NIST SP 800-57

H-02 [CRÍTICA] Logout no invalida refresh tokens ni limpia caché
────────────────────────────────────────────────────────────────────────────────

EVIDENCIA
  Archivo 1: apps/backend/api/src/app/auth/auth.controller.ts:251-257
  Código actual:
    @Post('logout')
    @HttpCode(HttpStatus.OK)
    @UseGuards(CsrfGuard)
    logout(@Res({ passthrough: true }) res: Response) {
      this.cookieService.clearAuthCookies(res);
      return { message: 'Logout exitoso' };
    }

  Archivo 2: apps/backend/api/src/app/auth/services/session.service.ts:325-329
  Código actual:
    async terminateAllSessions(userId: string) {
      await this.userCacheService.clearUserSession(userId);
      // Optional: Revoke all refresh tokens in DB if stricter security is needed
      // await this.refreshTokenRepository.update({ userId, isRevoked: false }, { isRevoked: true, revokedAt: new Date() });
    }

EXPLICACIÓN
  Dos bugs separados que se refuerzan:

  1. El endpoint POST /auth/logout NUNCA llama a authService.logout(userId). Solo
     llama a clearAuthCookies(). Resultado: el caché de sesión en Redis no se limpia
     y los refresh tokens en la base de datos no se revocan.

  2. Incluso si se llamara a authService.logout(), la línea que revoca los refresh
     tokens en la DB está COMENTADA como "opcional". No lo es: es un requisito de
     seguridad fundamental.

  Impacto: Un atacante que capture el refresh token (XSS, network interception,
  device compromise) puede continuar obteniendo nuevos access tokens después del
  logout del usuario legítimo, indefinidamente hasta que el token expire (7 o 30
  días según rememberMe).

CORRECCIÓN
  // auth.controller.ts
  @Post('logout')
  @HttpCode(HttpStatus.OK)
  @UseGuards(JwtAuthGuard, CsrfGuard)  // Añadir JwtAuthGuard para tener userId
  async logout(
    @CurrentUser() user: User,
    @Res({ passthrough: true }) res: Response,
    @Req() req: Request
  ) {
    // 1. Obtener el ID del refresh token actual de la cookie para preservarlo si se necesita
    const refreshToken = req.cookies?.refresh_token;

    // 2. Revocar tokens en DB y limpiar caché
    await this.authService.logout(user.id);

    // 3. Limpiar cookies
    this.cookieService.clearAuthCookies(res);
    return { message: 'Logout exitoso' };
  }

  // session.service.ts — descomentar y hacer obligatoria la revocación:
  async terminateAllSessions(userId: string) {
    await this.userCacheService.clearUserSession(userId);
    await this.refreshTokenRepository.update(
      { userId, isRevoked: false },
      { isRevoked: true, revokedAt: new Date() }
    );
  }

REFERENCIA
  OWASP Session Management Cheat Sheet (Session Termination),
  CWE-613 (Insufficient Session Expiration)

  activos son directamente utilizables.

  random hasheado (mismo patrón que UsersService):

  async sendPasswordResetLink(dto: ForgotPasswordDto) {

    const hashedToken = crypto.createHash('sha256').update(rawToken).digest('hex');
    user.security.passwordResetToken = hashedToken;  // Hash en DB
    user.security.passwordResetExpires = new Date(Date.now() + 15 * 60 * 1000);
    await this.userRepository.save(user);
    await this.mailService.sendPasswordResetEmail(user, rawToken, '15m');
  }

  async resetPassword(dto: ResetPasswordDto) {
    const { token, password } = dto;
    const hashedToken = crypto.createHash('sha256').update(token).digest('hex');
    const user = await this.userRepository.findOne({
      where: { 'security.passwordResetToken': hashedToken, ... }  // Buscar por hash
    });
    // Comparación de hash ya fue hecha en la query — no se necesita !=
  }

H-14 [MEDIA] Detección de viaje imposible bloquea login, demasiado agresiva
────────────────────────────────────────────────────────────────────────────────

EVIDENCIA
  Archivo: apps/backend/api/src/app/auth/services/security-analysis.service.ts:55-68
  Código actual:
    const maxSpeed = AuthConfig.IMPOSSIBLE_TRAVEL_MAX_SPEED;  // 1500 km/h (auth.config.ts:63)
    if (distanceKm > minDistance && speed > maxSpeed) {
      this.logger.warn(`[SECURITY] Impossible Travel Detected...`);
      throw new UnauthorizedException(
        'Viaje imposible detectado. Por seguridad, su cuenta ha sido bloqueada temporalmente...'
      );
    }

EXPLICACIÓN
  Dos problemas:

  1. 1500 km/h apenas supera la velocidad de los aviones comerciales (~900 km/h)
     con muy poco margen. Un usuario que viaja en avión con conexión a internet
     podría activar esto accidentalmente.

  2. La respuesta es un bloqueo completo del login (UnauthorizedException). Las
     mejores prácticas (Google, GitHub, Stripe) usan detección de viaje imposible
     para REQUERIR verificación adicional (código por email, confirmar dispositivo),
     no para bloquear outright. Un bloqueo puede ser un vector de DoS contra
     usuarios que usan VPN o servicios proxy.

CORRECCIÓN
  La práctica de la industria es:
  - Bajo: ignorar (mismo país, diferente ciudad)
  - Medio: requerir verificación por email / 2FA
  - Alto (velocidades realmente imposibles, > 3000 km/h): bloquear temporalmente
    y notificar

  // security-analysis.service.ts
  async checkImpossibleTravel(userId: string, currentIp?: string) {
    // ...cálculo existente...

      this.logger.warn(`[SECURITY] Suspicious Travel - ${speed} km/h`);
      // Emitir evento para notificación al usuario, no lanzar excepción
      this.eventEmitter.emit('security.suspicious_travel', { userId, speed, distanceKm });
      // No lanzar UnauthorizedException — dejar que el login continúe con alerta
    }
  }

REFERENCIA
  NIST SP 800-63B §6.1, Auth0 Anomaly Detection documentation

H-17 [BAJA] changePassword en frontend usa IS_PUBLIC_API=true incorrecto
────────────────────────────────────────────────────────────────────────────────

EVIDENCIA
  Archivo: apps/core/client-web/src/app/core/services/auth.ts:521-526
  Código actual:
    changePassword(data: any): Observable<{ message: string }> {
        return this.http.post<{ message: string }>(`${this.apiUrl}/change-password`, data, {
            context: new HttpContext().set(IS_PUBLIC_API, true)  // ← Incorrecto
        }).pipe(...)
    }

EXPLICACIÓN
  IS_PUBLIC_API=true indica al interceptor de auth que no intente refrescar el
  token si recibe un 401. Para un endpoint protegido como change-password, esto
  significa que si el access token expira exactamente durante la operación, el
  usuario recibirá un error sin que el interceptor intente renovar el token
  primero. Debería ser false (comportamiento por defecto).

CORRECCIÓN
  // auth.ts — eliminar el context o no pasar IS_PUBLIC_API
  changePassword(data: any): Observable<{ message: string }> {
    return this.http.post<{ message: string }>(`${this.apiUrl}/change-password`, data)
      .pipe(catchError(...));
  }

REFERENCIA
  Inconsistencia interna del proyecto
