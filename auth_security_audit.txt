AUDITORÍA DE SEGURIDAD DE AUTENTICACIÓN Y AUTORIZACIÓN — VIRTEX
Fecha de auditoría: 2026-06-01
Alcance: exclusivamente autenticación, autorización, usuarios, roles, tokens, sesiones, cookies, MFA, OAuth/WebAuthn y protección de rutas frontend/backend.
Benchmark usado: prácticas de Auth0/GitHub/Stripe, OWASP ASVS 4.0.3, OWASP Top 10 2021, OWASP Session Management Cheat Sheet, OWASP CSRF Prevention Cheat Sheet, OWASP Authentication Cheat Sheet, CWE, RFC 7519 y NIST SP 800-63B.

================================================================================
1. RESUMEN EJECUTIVO
================================================================================

Veredicto directo: NO está listo para producción a escala empresarial.

El repositorio contiene varias decisiones sólidas: cookies httpOnly para tokens, expiraciones cortas, hash de refresh tokens en base de datos, tokenVersion para invalidación global, validación global con whitelist y guards globales de JWT/throttling. Sin embargo, hay defectos de severidad alta/crítica que un equipo de seguridad tipo Auth0/GitHub/Stripe bloquearía antes de producción: el backend tiene un error de compilación por redeclaración de const en main.ts; el logout no invalida access tokens ya emitidos; el flujo OAuth de Microsoft auto-vincula cuentas existentes sin verificar explícitamente email_verified; la rotación de refresh tokens no es transaccional ni usa bloqueo de fila; el double-submit CSRF no está firmado; y operaciones sensibles como habilitar 2FA no exigen reautenticación fuerte.

La arquitectura muestra intención enterprise, pero todavía mezcla controles robustos con huecos críticos de sesión, MFA y OAuth. La puntuación global asignada es 6.1/10: buena base técnica, insuficiente para un entorno empresarial real hasta corregir las fases 1 y 2 de la hoja de ruta.

6. HOJA DE RUTA DE REMEDIACIÓN PRIORIZADA
================================================================================

FASE 1 — Crítico (antes de cualquier deploy a producción)

1. Corregir redeclaración de corsOrigins en main.ts. Esfuerzo: Bajo.
2. Cambiar terminateAllSessions/logout/forceLogout/remove user para incrementar tokenVersion o usar session-id denylist verificable. Esfuerzo: Medio.
3. Bloquear auto-link OAuth inseguro: exigir email_verified/issuer/audience/tenant para Microsoft y step-up para vinculación de cuenta existente. Esfuerzo: Medio.
4. Eliminar access_token del body en impersonate/stop-impersonation y actualizar tipos frontend. Esfuerzo: Bajo.
5. Proteger refresh-token rotation con transacción + pessimistic_write/serializable y pruebas de concurrencia. Esfuerzo: Alto.
6. Exigir reautenticación/TwoFactorVerifiedGuard/currentPassword o WebAuthn assertion para habilitar 2FA. Esfuerzo: Medio.

FASE 2 — Importante (próximo sprint)

1. Migrar CSRF a signed double-submit ligado a sesión o token synchronizer server-side. Esfuerzo: Medio.
2. Añadir contadores de intentos por LOGIN_2FA, TOTP window y tempToken; bloquear/destruir desafío tras N fallos. Esfuerzo: Medio.
3. Cambiar reset-password para no depender de query params persistentes; usar fragment o cookie httpOnly de reset tras consume. Esfuerzo: Medio.
4. Hacer logout frontend no silencioso cuando falla backend; agregar retry idempotente. Esfuerzo: Bajo.
5. Normalizar DTOs de auth/users con MaxLength, enums y DTOs para bodies directos (@Body('code')). Esfuerzo: Medio.
6. Configurar SCA en CI (npm audit/OSV/Snyk/Dependabot) y política de bloqueo high/critical. Esfuerzo: Bajo.

FASE 3 — Calidad y escalabilidad (deuda técnica planificada)

1. Añadir permissionsGuard/data.permissions en rutas frontend sensibles y mapa de permisos por módulo. Esfuerzo: Medio.
2. Reducir PII en auditoría/logs con masking/hash/cifrado y política de retención. Esfuerzo: Medio.
3. Separar claramente contratos browser DTO: nunca incluir accessToken/refreshToken en interfaces cliente. Esfuerzo: Bajo.
4. Añadir pruebas e2e de auth: logout invalida access token, refresh replay concurrente, OAuth linking, CSRF signed token, MFA brute-force. Esfuerzo: Alto.
5. Documentar matriz de roles/permisos y automatizar verificación de endpoints vs permisos. Esfuerzo: Alto.

CONCLUSIÓN
================================================================================

Hay una base seria y varios controles correctos, especialmente httpOnly cookies, tokenVersion, refresh-token hashing, whitelist global y guards por defecto. Pero el sistema aún no cumple un estándar empresarial comparable a Auth0/GitHub/Stripe por defectos de sesión, OAuth, MFA, CSRF y consistencia contractual. La recomendación objetiva es bloquear producción hasta completar FASE 1 y programar FASE 2 inmediatamente después.
