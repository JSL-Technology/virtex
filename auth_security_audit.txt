======================================================================
1. RESUMEN EJECUTIVO
======================================================================

El repositorio contiene una implementación de autenticación con varios controles empresariales reales: cookies httpOnly para tokens, refresh tokens rotados y persistidos, tokenVersion para invalidación global, Argon2, validación global de DTOs, CSRF signed double-submit, RBAC con permisos y protección de rutas frontend. Sin embargo, no está listo para producción a escala empresarial porque existen debilidades relevantes en mutaciones de perfil sin CSRF, cambio de email de perfil sin step-up ni invalidación de sesión, configuración incompleta de secretos JWT de refresh/reset, un flujo de refresh con ventana de gracia que puede permitir continuidad a un token robado reciente, enumeración en WebAuthn, logging de PII en eventos de seguridad y una inconsistencia funcional en el reseteo administrativo de contraseña.

El veredicto directo es: NO listo para producción empresarial sin remediación previa. La arquitectura base es recuperable y muestra intención de diseño seguro, pero los hallazgos de CSRF, gestión de email/sesión y secretos degradan el estándar por debajo de lo que esperaría de Auth0, GitHub, Stripe o Google para endpoints sensibles.

H-01 — ALTA — PATCH /users/profile carece de CsrfGuard
Tipo: Vulnerabilidad, CWE-352 Cross-Site Request Forgery
Capa: Backend

Evidencia actual:
Archivo: apps/backend/api/src/app/users/users.controller.ts:100-110
  @Patch('profile')
  @ApiOperation({ summary: 'Update current user profile' })
  async updateProfile(
    @CurrentUser() user: User,
    @Body() updateProfileDto: UpdateProfileDto,
  ) {
    const updatedUser = await this.usersService.updateProfile(user.id, updateProfileDto);
    return plainToInstance(UserResponseDto, updatedUser, { excludeExtraneousValues: true });
  }

Archivo relacionado: apps/backend/api/src/app/auth/services/cookie.service.ts:24-35
  const cookieOptions = { httpOnly: true, secure: isProduction, sameSite: 'lax', path: '/' };
  res.cookie('access_token', accessToken, { ...cookieOptions, maxAge: ... });

Explicación:
La autenticación se basa en cookies; por tanto, cualquier endpoint mutador que acepte cookies debe validar CSRF salvo que use SameSite=Strict y no haya flujos cross-site relevantes. SameSite=Lax reduce pero no elimina todas las clases de CSRF y no reemplaza tokens CSRF para mutaciones autenticadas. El endpoint permite cambiar datos de identidad/perfil sin CsrfGuard.

Corrección propuesta:
  @Patch('profile')
  @UseGuards(CsrfGuard)
  @ApiOperation({ summary: 'Update current user profile' })
  async updateProfile(...) { ... }

Referencia:
OWASP CSRF Prevention Cheat Sheet; OWASP ASVS 4.0.3 V3.5; CWE-352.

H-02 — ALTA — Cambio de email propio sin step-up, sin CSRF y sin invalidación de sesión
Tipo: Vulnerabilidad de autenticación/sesión, CWE-613 Insufficient Session Expiration
Capa: Backend

Evidencia actual:
Archivo: apps/backend/api/src/app/users/dto/update-profile.dto.ts:22-24
  @IsOptional()
  @IsEmail()
  email?: string;

Archivo: apps/backend/api/src/app/users/users.service.ts:40-51
  if (updateProfileDto.email && updateProfileDto.email !== user.email) {
    user.isEmailVerified = false;
  }
  Object.assign(user, updateProfileDto);
  await this.userCacheService.clearUserSession(id);

Explicación:
El email es un identificador primario de cuenta y suele participar en recuperación de contraseña, notificaciones de seguridad y login. Permitir modificarlo desde PATCH /profile sin contraseña actual, OTP/step-up ni incremento de tokenVersion deja una sesión robada capaz de tomar control persistente de la cuenta.     @IsString() @IsNotEmpty() currentPassword: string;
    @IsString() @IsOptional() otpCode?: string;
  }

  @Post('profile/email-change')
  @UseGuards(JwtAuthGuard, CsrfGuard, TwoFactorVerifiedGuard)
  async requestEmailChange(@CurrentUser() user, @Body() dto: RequestEmailChangeDto) {
    await this.usersService.requestEmailChange(user.id, dto);
    return { message: 'Confirma el cambio desde el correo nuevo.' };
  }

En el servicio, verificar contraseña, confirmar el nuevo correo por token de un solo uso, marcar isEmailVerified=false hasta confirmación e incrementar tokenVersion al completar:
  user.email = dto.newEmail.toLowerCase().trim();
  user.isEmailVerified = false;
  user.security.tokenVersion = (user.security.tokenVersion || 0) + 1;
  await this.userCacheService.clearUserSession(user.id);

Referencia:
OWASP Authentication Cheat Sheet: reauthentication for sensitive features; OWASP ASVS V2.2/V3.3; CWE-613.

H-04 — ALTA — Reuso de refresh token revocado dentro de ventana de gracia emite otro token
Tipo: Vulnerabilidad de sesión, CWE-613
Capa: Backend

Evidencia actual:
Archivo: apps/backend/api/src/app/auth/services/session.service.ts:116-152 y 190-197
  if (!refreshTokenEntity || refreshTokenEntity.isRevoked) {
    const GRACE_PERIOD = AuthConfig.REFRESH_GRACE_PERIOD;
    if (refreshTokenEntity?.revokedAt && Date.now() - refreshTokenEntity.revokedAt.getTime() < GRACE_PERIOD) {
      this.logger.warn(`[SECURITY] Refresh token reused within grace period. Issuing new token.`);
      if (refreshTokenEntity.replacedByToken) {
        await this.refreshTokenRepository.update(refreshTokenEntity.replacedByToken, { isRevoked: true, revokedAt: new Date() });
      }
    } else { ... family invalidation ... }
  } else {
    refreshTokenEntity.isRevoked = true;
    refreshTokenEntity.revokedAt = new Date();
    await this.refreshTokenRepository.save(refreshTokenEntity);
  }
  const authResponse = await this.tokenService.generateAuthResponse(user, {}, ipAddress, sanitizedUserAgent);

Explicación:
Si un atacante roba un refresh token y lo usa pocos segundos después de que el usuario legítimo lo rotó, el código entra en la gracia, revoca el token reemplazo legítimo y después genera otro token para quien presentó el token viejo. La gracia resuelve carreras de cliente, pero implementada así también premia al portador de un token revocado reciente.

Corrección propuesta:
No emitir token nuevo para un token ya revocado. Para manejar carreras legítimas, devolver el replacedByToken existente solo si se puede probar misma request id/dispositivo y nunca revocar al sucesor por defecto; preferible usar family invalidation si hay reuse fuera de una transacción atómica.

Ejemplo:
  if (!refreshTokenEntity || refreshTokenEntity.isRevoked) {
    await this.invalidateTokenFamily(user.id);
    throw new UnauthorizedException(AuthError.REFRESH_TOKEN_REVOKED);
  }
  await this.dataSource.transaction(async manager => {

CONCLUSIÓN FINAL
======================================================================

El módulo de autenticación tiene fundamentos serios y varias decisiones correctas: cookies httpOnly, CSRF firmado, Argon2, rotación de refresh tokens, tokenVersion, RBAC, cache con circuit breaker y route guards. No obstante, bajo estándares empresariales, los defectos de mutaciones sin CSRF, cambio de email sin reautenticación/invalidation, secretos incompletamente validados y refresh reuse impiden aprobarlo para producción. La recomendación es bloquear despliegue productivo hasta completar Fase 1 y planificar Fase 2 como siguiente sprint de seguridad.
