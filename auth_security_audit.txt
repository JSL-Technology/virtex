AUDITORÍA DE SEGURIDAD DE AUTENTICACIÓN Y AUTORIZACIÓN — Virteex
Fecha de auditoría: 2026-06-01
Alcance: frontend y backend, exclusivamente autenticación, autorización, usuarios, roles, tokens y seguridad de sesiones.
Benchmark declarado: prácticas de equipos tipo Google, GitHub, Stripe, Auth0; referencias OWASP ASVS / Cheat Sheets / API Top 10, CWE, RFC 6265bis, RFC 7519, NIST SP 800-63B y NIST SP 800-57.

================================================================================
1. RESUMEN EJECUTIVO
================================================================================

El módulo de autenticación tiene una base considerablemente mejor que un MVP típico: usa cookies httpOnly para tokens, CSRF firmado, Argon2, refresh tokens hasheados en base de datos, rotación con bloqueo pesimista, invalidación por tokenVersion, filtros de DTO y controles RBAC en rutas clave. Sin embargo, NO está listo para producción a escala empresarial porque aún hay fallos de diseño que un equipo de seguridad de Auth0/GitHub/Stripe no aceptaría: reCAPTCHA se desactiva fuera de producción, hay secretos críticos no validados centralmente al arranque, existe un endpoint de invitación con token en path que puede filtrarse por logs/referrers, algunas rutas frontend de configuración quedan visibles sin guard granular, el flujo de 2FA expone el temp token al JavaScript y algunos errores administrativos permiten enumeración dentro de organizaciones. El backend está mejor que el frontend en protección real, pero hay incoherencias y dependencias de configuración que pueden romper controles en despliegues multiinstancia o entornos no productivos expuestos. El veredicto directo es: apto para continuar endurecimiento, NO apto para producción empresarial sin remediar Fase 1 y Fase 2.

H-01 — [ALTA] Validación central de secretos incompleta y fallback inconsistente.

Evidencia: código actual
Archivo: apps/backend/api/src/app/app.module.ts:70-95
Fragmento:

    DB_HOST: Joi.string().required(),
    ...
    RECAPTCHA_V3_SECRET_KEY: Joi.string().required(),
    AWS_S3_BUCKET_NAME: Joi.string().required(),
    ...
    STRIPE_WEBHOOK_SECRET: Joi.string().required(),
  });

Archivo: apps/backend/api/src/app/auth/services/token.service.ts:41-47
Fragmento:
  const secret = this.configService.get<string>('ENCRYPTION_SECRET');
  const salt = this.configService.get<string>('AUTH_SALT');
  const effectiveSecret = secret || 'default-secret-change-me-in-prod-32';
  const effectiveSalt = salt || 'default-salt-change-me-in-prod';
  this.encryptionKey = crypto.scryptSync(effectiveSecret, effectiveSalt, 32);

Archivo: apps/backend/api/src/app/auth/services/session.service.ts:59-85
Fragmento:
  if (!secret) { if (isProduction) { throw new Error('FATAL: ENCRYPTION_SECRET is not defined in production environment.'); } }
  if (isProduction) { if (!salt || salt === 'default-salt-change-me-in-prod') { throw new Error('FATAL: AUTH_SALT is not defined or is using default value in production.'); } }

Explicación:
La validación Joi no declara secretos de JWT, refresh, CSRF, 2FA, preverify, encryption ni salt. Algunas rutas fallan tarde con getOrThrow (JwtStrategy), otras aceptan fallback en desarrollo o incluso derivan una key default en TokenService. En arquitectura enterprise, la app debe fallar rápido y de forma uniforme ante secretos ausentes/débiles, y debe tener separación/rotación documentada.

Corrección recomendada:
  const requiredSecret = Joi.string().min(32).required();
  const envValidation = Joi.object({
    NODE_ENV: Joi.string().valid('development', 'test', 'production').default('development'),
    JWT_SECRET: requiredSecret,
    JWT_REFRESH_SECRET: requiredSecret.invalid(Joi.ref('JWT_SECRET')),
    JWT_2FA_TEMP_SECRET: requiredSecret,
    JWT_PREVERIFY_SECRET: requiredSecret,
    CSRF_SECRET: requiredSecret,
    ENCRYPTION_SECRET: requiredSecret,
    AUTH_SALT: Joi.string().min(16).required(),
    ...
  });

Además, TokenService debe replicar la política fail-fast de SessionService o consumir una KeyManagementService única:
  onModuleInit() {
    const secret = this.configService.getOrThrow<string>('ENCRYPTION_SECRET');
    const salt = this.configService.getOrThrow<string>('AUTH_SALT');
    if (process.env.NODE_ENV === 'production' && /change-me|default/i.test(secret + salt)) {
      throw new Error('FATAL: weak encryption configuration');
    }
    this.encryptionKey = crypto.scryptSync(secret, salt, 32);
  }

Referencia: OWASP ASVS V14 Configuration, OWASP Secrets Management Cheat Sheet, NIST SP 800-57 key separation/management, CWE-798/CWE-321/CWE-16.
