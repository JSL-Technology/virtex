1. RESUMEN EJECUTIVO

El módulo de autenticación tiene componentes fuertes y modernos: cookies httpOnly para tokens, rotación de refresh tokens con hash en base de datos, versionado de tokens para invalidación global, Argon2 para contraseñas, validación global de DTOs, rate limiting global, reCAPTCHA en login/registro/forgot-password y CSRF explícito en varias operaciones sensibles. Sin embargo, no está listo para producción empresarial al nivel de Auth0/GitHub/Stripe porque contiene fallas de autorización y sesión que sí soportan escalación de privilegios o bypass de controles: creación/edición de roles acepta permisos arbitrarios incluyendo '*', varios endpoints mutantes de usuarios carecen de CsrfGuard, endpoints de admin no filtran organización en force-logout/blockAndLogout, OAuth no evidencia state/nonce y el refresh de sesión usa GET público con cookies. Además, hay inconsistencias de permisos ('users.create' vs 'users:create') que rompen controles de usuario y diferencias frontend/backend que pueden dejar la sesión en estado inconsistente tras refresh. El diseño es prometedor, pero requiere remediación estructural antes de cualquier despliegue productivo con datos reales.

Veredicto directo: NO está listo para producción a escala empresarial. La razón principal no es ausencia total de seguridad, sino controles críticos implementados de forma incompleta o inconsistente en rutas de administración de identidad, RBAC, OAuth y CSRF.

```
54     }
```
- apps/backend/api/src/app/shared/permissions.ts:112
```

```

Explicación:
El backend expone CRUD de roles y solo valida que permissions sea un array; no valida que cada permiso pertenezca a ALL_PERMISSIONS. Como el util y el guard tratan '*' como super-admin, cualquier actor con capacidad de crear/editar roles puede introducir '*' u otros permisos inventados. Si además puede asignar ese rol, o edita un rol que ya posee, obtiene autorización universal. Esto no resistiría revisión en Auth0/GitHub/Stripe: los permisos son datos de alta criticidad y deben ser de catálogo cerrado, con workflows de aprobación y auditoría.

Corrección propuesta:
```
import { ArrayNotEmpty, IsArray, IsIn, IsString } from 'class-validator';
import { ALL_PERMISSIONS, Permission } from '../../shared/permissions';

export class CreateRoleDto {
  @IsString()
  @Length(3, 100)
  name: string;

  @IsArray()
  @ArrayNotEmpty()
  @IsString({ each: true })
  @IsIn(ALL_PERMISSIONS, { each: true })
  permissions: Permission[];
}
```
Además:
- Eliminar '*' del modelo general o restringirlo a un rol seed super-admin no editable por API.
- En RolesService.update/create, rechazar '*' explícitamente salvo proceso interno firmado.
- Agregar auditoría obligatoria para cambios de permisos.
- Agregar tests de integración: roles:create con ['*'] debe devolver 400/403.

Referencia:
- OWASP Authorization Cheat Sheet: deny by default, validar permisos en cada request y justificar permisos concedidos.
- CWE-269 / CWE-266 (gestión incorrecta de privilegios) y CWE-639 en contexto de bypass de autorización.

Explicación:

Corrección propuesta:
```

@HasPermission(PERMISSIONS.USERS_CREATE)

@Patch('profile')
@UseGuards(CsrfGuard)
async updateProfile(...) { ... }

@Patch(':id')
@UseGuards(CsrfGuard, TwoFactorVerifiedGuard)
@HasPermission(PERMISSIONS.USERS_EDIT)
async update(...) { ... }

@Delete(':id')
@UseGuards(CsrfGuard, TwoFactorVerifiedGuard)
@HasPermission(PERMISSIONS.USERS_DELETE)
async remove(...) { ... }
```
Mejor aún: aplicar CsrfGuard globalmente a métodos unsafe autenticados y usar @SkipCsrf solo en endpoints públicos con proof-of-possession.

Referencia:
- OWASP CSRF Prevention Cheat Sheet: las operaciones que cambian estado con cookies requieren tokens anti-CSRF o defensas equivalentes.

6. HOJA DE RUTA DE REMEDIACIÓN PRIORIZADA

FASE 1 — Crítico (antes de cualquier deploy a producción)

1. Bloquear permisos arbitrarios en roles (Esfuerzo: Medio)
   - Validar permissions contra ALL_PERMISSIONS con `each: true`.
   - Rechazar '*' por API o restringirlo a proceso interno.
   - Agregar tests de create/update role con permisos inválidos.

2. Corregir permisos users.* a PERMISSIONS.USERS_* (Esfuerzo: Bajo)
   - Reemplazar strings hardcodeados en UsersController por constantes.
   - Testear acceso con roles seed reales.

3. Aplicar CSRF a todas las mutaciones autenticadas de usuarios (Esfuerzo: Medio)
   - invite, profile patch, update user, delete user, avatar upload si aplica.
   - Considerar CsrfGuard global para métodos unsafe.

4. Arreglar IDOR de forceLogout/blockAndLogout (Esfuerzo: Bajo)
   - Pasar organizationId desde CurrentUser.
   - Filtrar `{ id, organizationId }` en service.

5. Cambiar refresh de GET a POST + CSRF/Origin checks (Esfuerzo: Medio)
   - Actualizar frontend interceptor/AuthService.
   - Mantener path restringido del refresh cookie.

FASE 2 — Importante (próximo sprint)

1. OAuth/OIDC hardening (Esfuerzo: Alto)
   - state, nonce, PKCE donde aplique.
   - Validar issuer/audience/email_verified/tenant.
   - Revisión de account linking.

2. Step-up MFA para IAM admin (Esfuerzo: Medio)
   - Invite, update role assignment, delete user, force logout, password reset admin.
   - Requerir autenticación reciente o OTP por acción.

3. JWT hardening y rotación de claves (Esfuerzo: Alto)
   - Algoritmo explícito, issuer/audience, kid.
   - Separación access/refresh secrets ya existe parcialmente; formalizar rotación.

4. Unificar contrato frontend/backend (Esfuerzo: Medio)
   - Generar cliente desde Swagger/OpenAPI.
   - Corregir refresh response typing y users API drift.

5. Minimización de PII en logs (Esfuerzo: Medio)
   - Hashear emails en logs operacionales.
   - Mantener audit trail protegido y con retención.

FASE 3 — Calidad y escalabilidad (deuda técnica planificada)

1. Prefijos de cookies `__Host-`/`__Secure-` (Esfuerzo: Bajo)
2. Aumentar MaxLength de password y agregar breached-password screening (Esfuerzo: Medio)
3. No-store headers en endpoints de sesión/auth (Esfuerzo: Bajo)
4. Tests e2e de seguridad auth/RBAC/CSRF/IDOR (Esfuerzo: Alto)
5. Auditoría estructurada de cambios de roles/permisos con before/after (Esfuerzo: Medio)

CONCLUSIÓN FINAL

El repositorio muestra un esfuerzo serio por construir auth moderno: cookies httpOnly, Argon2, refresh rotation, tokenVersion, cache, CSRF parcial, 2FA y passkeys. No obstante, los controles están incompletos en los lugares más delicados: roles/permisos, endpoints de administración de usuarios, OAuth y consistencia de sesión. La recomendación es bloquear release productivo hasta completar FASE 1 y aceptar solo entornos internos controlados para pruebas. Con FASE 1 + FASE 2, el módulo podría acercarse a un estándar enterprise defendible.
